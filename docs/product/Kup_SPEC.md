## 0. 용어

- **채널**: 서비스가 운영을 돕는 사용자의 인스타 비즈니스 계정 1개.
- **온보딩 설문**: 사람·계정·톤·금지표현·발행 빈도를 시작 시 수집해 계정 컨셉·전략·추천 주제를 도출하는 단계.
- **계정 컨셉**: 설문 기반으로 1회 확정하는 페르소나·톤·비주얼·콘텐츠 방향. 이후 모든 생성의 제약으로 상속된다.
- **AI 기획 리스트**: 워크스페이스의 콘텐츠 기획 단위 테이블(형식·장수·주제·본문 구성). 각 행이 카드뉴스 1건의 기획.
- **시안(초안)**: AI가 생성한 카드뉴스 초안. **항상 편집 가능**하며, 검수 게이트 통과 전에는 발행 불가.
- **검수 게이트**: 발행·내보내기 전 반드시 통과하는 필수 관문(민감/권유 표현, 표기 누락, 미검증 주장 플래그 + AI 라벨 확인 + 사용자 최종 승인).
- **휴먼인더루프(코파일럿)**: 생성은 AI가 가속하되, **최종 검수·승인·발행 주도권은 항상 사용자**에게 있는 운영 방식.
- **리드마그넷**: 댓글 트리거 → 정보 DM 발송으로 잠재 팔로워를 모으는 공식 API 기반 그로스 기능.
- **3-레이어**: 생성(Claude) / 발행·데이터(Meta Graph API) / 오케스트레이션(BullMQ+Redis 작업큐 — 예약·cron·재시도).

## 1. Proposal — 왜 만드는가

### 1.1 문제
인스타를 갓 시작한 1인 인플루언서는 수익화 진입선인 **팔로워 1,000명**을 첫 목표로 삼지만, 대다수는 **꾸준한 발행**이라는 첫 관문에서 멈춘다. 세 가지 마찰 때문이다.

1. **속도(Velocity)** — 카드뉴스 한 건에 2~3시간, 검색탭→피그마/캔바/미리캔버스→캡컷으로 툴을 계속 갈아탄다. 제작 부담에 업로드 리듬이 무너진다.
2. **신뢰(Trust)** — AI 산출물이 내 톤·주체성과 어긋나 재작업이 생기고, 무승인 자동게시는 불안해 꺼버린다. AI를 켜도 손이 덜 가지 않는다.
3. **운영 지속성(Operation)** — 성과 데이터는 보지만 "다음에 뭘 바꿀지"가 체계화되지 않아 성장이 정체된다.

### 1.2 핵심 인사이트
- 빈자리는 **더 많은 기능도, 완전 자동화도 아니다.** 도구는 이미 많지만 흩어져 있고, 어느 것도 한 사람의 계정을 *이어서* 함께 관리해주지 않는다. 반대로 완전 자동화에는 "내 계정은 내가 만든다"는 거부감이 따른다.
- 따라서 포지셔닝은 **코파일럿**: ① 더 빨리 만들고(가속) ② 한곳에서 기획·제작·발행하고(통합) ③ 다음 액션으로 연결해 관리한다(운영). **검수·발행 주도권은 항상 사용자**에게 둔다.
- 플랫폼 역풍 대응: 인스타는 자동화·양산형 콘텐츠 도달을 낮추고 AI 라벨을 강제하는 추세. 휴먼인더루프 + AI 라벨 + 체계적 검수를 약점이 아닌 **신뢰·채택 장치**로 쓴다.

### 1.3 목표
인스타를 갓 시작한 1인 인플루언서가 **카드뉴스를 꾸준히 만들어 직접 발행하도록** 만들어, 1K 도달까지 운영을 지속시킨다. 한국 우선·테스터 한정 베타로 검증한다.

## 2. Design 결정 (Architecture & Key Decisions)

### 2.1 3-레이어 분리 (핵심 결정)
| 레이어 | 책임 | 도구 |
|---|---|---|
| 생성 | 기획·카피·캡션·해시태그 등 **언어 가공만** | Claude API |
| 발행·데이터 | 인스타 게시·성과 수집 | Meta Graph API (앱 검수 필수) |
| 오케스트레이션 | 예약·발행·웹훅·재시도 | BullMQ + Redis (작업큐·cron) |

**근거**: Claude API로는 인스타 직접 접근 불가. 세 레이어를 분리해야 각각 독립적으로 교체·확장할 수 있다. *(영상/릴스 렌더는 MVP 범위 밖 — 5장 참조.)*

### 2.2 인증/데이터
- IG 비즈니스 계정 **OAuth 연동**. 비밀번호 미보관, 토큰 암호화 저장.
- 성과는 Graph API로만 수집. 계정 단위(`followers_count`, `follows_and_unfollows`)와 게시물 단위(`reach`, `saved`, `shares`, `likes`, `comments`, `profile_visits`, `follows`).

### 2.3 발행 메커니즘
- Graph API 2단계 발행(컨테이너 생성 → FINISHED 확인 → 게시) 준수.
- 네이티브 예약이 없으므로 BullMQ 지연 잡이 예약 시각에 발행 워커를 실행한다.

### 2.4 콘텐츠 = 카드뉴스 단일 엔진 (단일 타깃)
- 타깃은 **"인스타를 갓 시작해 수익화 계정이 목표인 1인 인플루언서"** 하나. (구버전 A·B·C 3모드 폐기 — Decision Log 참조.)
- 전제: 타깃은 **이미 프로페셔널(비즈니스/크리에이터) 계정으로 전환 완료**한 상태. (발행·인사이트·메시지 API의 필수 전제 충족.)
- 산출물은 **카드뉴스**(페이지 구성·문구·캡션·해시태그·CTA). 형식은 카드뉴스 / 정사각형 카드뉴스, 장수 선택 가능.
- 릴스·숏츠·영상 증강은 MVP 밖(5장).

### 2.5 운영 모델 = 코파일럿 (휴먼인더루프 고정)
- **최종 검수·승인·발행은 항상 사용자.** 무검수 자동 발행(오토파일럿)은 MVP에서 제공하지 않는다. 신뢰가 핵심 가치이기 때문이다.
- AI는 생성을 가속할 뿐, 계정의 주인은 끝까지 사용자.

## 3. Requirements (Capabilities)

> 형식: `Requirement`(시스템이 반드시 보장할 것) + `Scenario`(WHEN/THEN). 구현·테스트의 기준. IA 매핑은 각 항목에 표기.

### 3.1 Capability: 온보딩 & 계정 컨셉 설계 *(IA 0.1.1.1 온보딩 / 0.2)*

#### Requirement: 온보딩 설문 → 컨셉·전략 자동 생성
시스템은 가입 직후 **온보딩 설문**(주제·톤·금지표현·발행 빈도, 타깃과 타깃의 결핍, 제공 가치)을 수집하고, 이를 바탕으로 **계정 컨셉과 추천 주제**를 자동 생성해야 한다(SHALL). 확정된 컨셉은 이후 모든 시안 생성의 제약으로 상속된다.

##### Scenario: 설문 완료 후 컨셉 수령
- WHEN 사용자가 온보딩 설문을 완료
- THEN 시스템은 계정 컨셉(예: "홈베이킹 초보 레시피")과 추천 주제 목록을 제시
- AND 추천 주제 선택 시 해당 컨셉·금지표현을 제약으로 카드뉴스 초안 생성에 사용

#### Requirement: IG 계정 연동 *(IA 0.2.1)*
시스템은 사용자의 IG 비즈니스 계정을 OAuth로 연동해야 하며(SHALL), 비밀번호를 저장하지 않고 토큰을 암호화 보관해야 한다. 연동 페이지에는 **연동 방법 가이드와 안전 안내 문구**를 함께 제공해야 한다(SHALL).

##### Scenario: 계정 연동 미완료
- WHEN 비즈니스 계정 전환·페이지 연결 등 전제 조건 미충족
- THEN 시스템은 무엇이 필요한지 안내하고 발행 기능을 잠금 상태로 유지

### 3.2 Capability: 카드뉴스 생성 (가속·통합) *(IA 0.2.3 AI 기획 리스트 → 제작·편집)*

#### Requirement: AI 기획 리스트
시스템은 콘텐츠 기획을 테이블(형식·카드뉴스 장수·주제·본문 타이틀+내용×장수)로 관리해야 하며(SHALL), 각 행에서 "제작하러 가기"로 카드뉴스 제작에 진입하게 해야 한다. 기획 행 추가가 가능해야 한다.

#### Requirement: 카드뉴스 초안 생성
시스템은 확정 컨셉과 선택 주제에 따라 **툴 전환 없이** 카드뉴스 초안(페이지 구성·문구·캡션·해시태그·CTA)을 한 번에 생성해야 한다(SHALL). 템플릿(+브랜드 컬러)과 정사각형 형식을 지원한다.

##### Scenario: 주제 선택 → 초안 생성
- WHEN 사용자가 추천 주제 중 하나를 선택
- THEN 시스템은 페이지 구성·문구·캡션·해시태그·CTA를 포함한 편집 가능한 카드뉴스 초안을 생성

#### Requirement: 편집 가능한 초안
시스템이 생성하는 모든 결과물은 **항상 편집 가능한 초안**이어야 한다(SHALL). 사용자는 문구·이미지를 자기 스타일로 수정할 수 있어야 한다.

### 3.3 Capability: 검수 게이트 (신뢰·주도권) — 차별화 핵심 *(IA 0.2.3.3.1 최종 검수 및 업로드)*

> 대원칙: **검수는 독립 기능/버튼이 아니라 발행 액션에 내장된 게이트다.** 통과 없이는 발행되지 않는다.
> 설계 의도: "제작→자동검수→편집→재검수" 상시 루프는 복잡해서 폐기. 편집은 자유, 검수는 **발행 시점에만** 자동 1회 실행.

#### Requirement: 발행 내장 검수 게이트
시스템은 별도 '검수' 단계/버튼을 두지 않고, **발행 버튼 클릭 시 검수 프로세스를 자동으로 1회 실행**해야 한다(SHALL). 제작 후 사용자는 자유롭게 편집하며, 편집 중에는 검수를 반복 강제하지 않는다. 발행 시 자동 검수는 다음을 수행한다:
1. **AI 자가 점검 / 플래그** — 출처 확인 · 민감 표현(권유·강요) 점검 · 표기 누락 체크. 플래그된 항목만 판단하게 한다.
2. **'AI 생성 콘텐츠' 라벨 확인** — 사용자가 편집을 거치면 명시 해제 규칙 적용.
3. **사용자 최종 승인/거절 분기.** 검수를 통과하지 않은 시안은 발행되지 않는다(MUST NOT).

##### Scenario: 발행 시 자동 검수
- WHEN 사용자가 발행 버튼을 클릭
- THEN 시스템은 내장 검수를 자동 실행하고, 플래그가 없으면 승인 후 발행, 플래그가 있으면 해당 항목을 표시
- AND 사용자가 수정 후 다시 발행을 누르면 검수가 재실행된다 (상시 편집-재검수 루프는 강제하지 않음)

#### Requirement: 휴먼인더루프 + 감사 추적 *(IA 0.2.3.3.1.1)*
시스템은 발행 직전 사용자 **승인/거절**을 강제하고(SHALL) 무승인 자동 발행을 제공하지 않으며(MUST NOT), 누가·언제·무엇을 승인했는지 승인 로그를 남긴다(감사 추적).

### 3.4 Capability: 콘텐츠 관리 (운영) *(IA 0.2.3.4 콘텐츠 관리 — 칸반)*

#### Requirement: 칸반 콘텐츠 보드
시스템은 게시물을 **칸반 형식 6단계(기획 중 / 기획 완료 / 제작 중 / 제작 완료 / 예약 업로드 / 업로드 완료)**로 관리해야 한다(SHALL). 캘린더/리스트 토글 뷰, 관련 게시물·임시저장 토글을 제공한다.

### 3.5 Capability: 발행 (BullMQ + Graph API) *(IA 0.2.3 / 예약 발행)*

#### Requirement: 승인 후 발행
시스템은 검수 게이트를 통과해 사용자가 승인한 시안만 발행해야 한다(SHALL). 발행은 Graph API 2단계(컨테이너 → FINISHED → 게시)를 거친다.

##### Scenario: 예약 발행
- WHEN 예약 시각 도달
- THEN BullMQ 워커가 발행 잡을 실행하고, 컨테이너가 FINISHED가 된 뒤 게시

### 3.6 Capability: 그로스 — 코멘트→DM 리드마그넷 *(IA 0.2.3.4 자동화 설정)*

#### Requirement: 리드마그넷
시스템은 공식 Messaging/Comments API로 리드마그넷을 운영해야 한다(SHALL): 지정 키워드 댓글 트리거 → 보낼 자료(정보) DM 발송. 자동팔로우·콜드 DM은 사용하지 않는다(MUST NOT).

#### Requirement: 요금제별 DM 발송 한도
시스템은 요금제 기준으로 DM 발송량을 차등 적용해야 한다(SHALL): **베이직 100건 / 프로 1,000건 / 프리미엄 제한 없음**. *(정확한 수치는 6장 결정 대기.)*

##### Scenario: 댓글 트리거
- WHEN 사용자가 지정 키워드를 댓글로 남김
- THEN 시스템은 공식 API로 해당 사용자에게 정보 DM을 발송하고 요금제 한도 내에서 자체 DM 로그에 기록

### 3.7 Capability: 인사이트 & 게이미피케이션 (운영 지속) *(IA 0.2.3.5)*

#### Requirement: 인스타 인사이트 대시보드
시스템은 Graph API로 성과를 수집·표시해야 한다(SHALL):
- **계정 단위**: 현재 총 팔로워 수(`followers_count`, 호출 시점값), 팔로워 유입·이탈 분리(`follows_and_unfollows` — 100명 이상부터 제공).
- **게시물 단위(발행 이후 누적값)**: 조회(`views`), 도달(`reach`), 저장(`saved`), 공유(`shares`), 좋아요(`likes`), 댓글(`comments`), 프로필 방문(`profile_visits`), 게시물 기여 팔로우(`follows`).
- **리드마그넷 자체 데이터**: 트리거·DM 발송·전환 카운트.
- 인사이트 업데이트 시각 표기(`YYYY.MM.DD 04:00:00` — 수치 안정 시간대 + 요청 부담 분산) + 새로고침 버튼.

#### Requirement: 발행 꾸준함 게이미피케이션 *(IA 0.2.3.5.1~2)*
시스템은 발행 꾸준함을 시각화해 반복 루틴을 유도해야 한다(SHALL):
- **팔로워 달성 챌린지** — 1칸=하루, 칸 농도=그날 발행 확정 건수(0·1·2·3·4+ 색 5단계). 발행 성공 시 자동 카운트, 예약 발행은 실제 발행된 날에 적립.
- **업로드 릴레이** — Contributions Graph(기여 그래프)로 표기.

#### Requirement: 다음 액션 인사이트
시스템은 성과에 기반해 **다음에 제작할 콘텐츠 방향**을 제안해야 한다(SHALL) — "보는 것"에서 "다음 행동"으로 연결.

### 3.8 Capability: 계정·요금제·결제 *(IA 0.2.2 / 0.3 / 0.4 / 0.5)*

#### Requirement: 회원가입·인증
시스템은 이메일·구글 연동 가입을 지원해야 하며(SHALL): 약관 동의(필수 이용약관·개인정보 / 선택 광고성, 전체 동의 일괄 체크), 이메일 형식·중복 확인, 비밀번호 규칙(숫자·영문·특수문자 8~16자·재확인 일치), 닉네임(10자 이내·중복 확인), 이메일 인증(링크/코드)을 제공한다. 구글 가입은 외부 계정 인증으로 이메일·비번을 직접 입력받지 않는다. 비밀번호 재설정은 이메일 가입자만 가능하다. 가입 완료 시 환영 화면 → 온보딩(워크스페이스)로 이동한다.

#### Requirement: 요금제 구독 & 결제
시스템은 요금제 3종(**베이직 / 프로 / 프리미엄**)을 제공해야 한다(SHALL): 기능·가격 비교표, **베타 기간 무료 운영**, 월/연 결제 토글(연간 할인 30%).
- **결제 플로우**: 플랜 선택·업그레이드 → 주문요약(플랜명·주기·금액·부가세·다음 결제일·쿠폰/프로모션) → 결제 정보 입력 → 결제 결과.
- **결제 수단**: PG사 연동(포트원/토스페이먼츠) + 간편결제(카카오·네이버·토스). **카드번호는 PG가 직접 수집하며 서비스는 보관하지 않는다(MUST NOT).** 자동결제 동의를 받는다.
- **구독 관리**: 결제수단 관리(등록·변경·삭제·대표 설정), 플랜 변경(업·다운그레이드), 자동결제 ON/OFF, 구독 해지(사유 설문·혜택 안내·이용 종료일 고지·재구독).
- **증빙**: 결제·환불 내역, 영수증·세금계산서·현금영수증(개인·사업자) 발급.
- 마이페이지에서 구독 정보·결제 내역(영수증 조회) 확인 및 데이터 삭제(전체 / 1·7·30일)를 제공한다.

### 3.9 Capability: 컴플라이언스 & 안전

#### Requirement: AI 라벨 & 감사 추적
시스템은 AI 생성 콘텐츠에 AI 라벨을 기본 적용하고(SHALL), 승인 로그(누가·언제·무엇)를 남겨야 한다.

#### Requirement: 도메인 컴플라이언스
시스템은 도메인 규칙을 검수 플래그에 주입해야 한다(SHALL): 금융·의료=권유·단정 표현 차단, 협찬·뒷광고 자동 표기.

> ⚠️ 출시 전 확인: '1,000명' 메시지가 보장으로 오해되지 않도록 커뮤니케이션 가드 필요. DM 리드마그넷의 스팸 신고·정책 위반 위험 모니터링.

## 7. Decision Log (왜 이렇게 정했나 — 재질문 방지용)

- **타깃 A·B·C 3모드 → 단일 타깃**: 세그먼트 동시 추진은 초점 분산. PRD 최종에서 "인스타 갓 시작 1인 인플루언서"로 단일화, 단일 카드뉴스 엔진으로 정리.
- **오토파일럿(무검수 자동흐름) 제거 → 휴먼인더루프 고정**: 신뢰가 핵심 가치이자 차별점. "최종 결재·발행은 사용자"가 PRD 대원칙. 자동 양산형은 플랫폼 알고리즘이 누르고 규제·신뢰 리스크도 큼.
- **카드뉴스 단일(릴스/숏츠 제외)**: MVP 제작 난이도·속도. 영상은 후속으로 미룸.
- **3-레이어 분리 유지**: Claude API로 인스타 직접 접근 불가. 생성/발행/오케스트레이션 독립 교체 위해.
- **공식 API만**: 자동팔로우·콜드DM은 계정 정지 직결. 성장은 코멘트→DM 등 합법 레버로만.
- **게이미피케이션(월별 챌린지) 추가**: 운영 지속성(세 번째 마찰) 해소 장치. "성과를 봐도 다음 액션이 없다"를 로드맵·루틴 트래커로 연결.
- **KPI를 PRD 최종 지표에 정합**: 구버전 North Star(주간 팔로워 순증) 중심에서, PRD가 명시한 발행 지속·승인·리드·만족도 중심 지표 세트로 재정렬.
- **오케스트레이션 n8n → BullMQ+Redis**: 예약·발행·웹훅·재시도는 제품의 핵심 비즈니스 로직(민감한 토큰·검수 상태 포함)이라 코드 기반 작업큐가 버전관리·테스트·에러추적·재시도 제어에 유리. n8n은 핵심 경로에서 빼고 초기 스파이크/비핵심 내부 자동화로만 선택 사용. 스파이크로 내구성 검증 후 `workers/`로 이식. 근거: [기술설계 ⑦](../tech/Kup_기술설계.md) · [n8n 결정](../tech/Kup_n8n_결정.md) · `spike-bullmq/`.
