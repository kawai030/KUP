# Kup — 온보딩 설문 설계 (계정 컨셉 도출)

> **문서 성격**: 가입 직후 받는 **온보딩 설문**의 문항·입력형식·필수여부 + **답 → 계정 컨셉 매핑**을 확정하는 문서. (백로그 1.1 + 1.3)
> **기준일**: 2026-06-26 · **선행**: [SPEC 3.1](Kup_SPEC.md) · [화면흐름 O1→O2](Kup_화면흐름.md) · [데이터모델 channel_configs](../tech/Kup_데이터모델.md) · [생성 파이프라인 §1.1 Concept](../tech/Kup_생성파이프라인_설계.md)
> **목표 산출**: 설문 답 → `channel_configs`(persona/tone/pillars/cadence/visual) + 금지표현 + 추천 주제 seed.

---

## 0. 설계 원칙

1. **마찰 최소화 vs 정보 충분** — 갓 시작한 1인 인플루언서가 중도이탈 않게 **핵심 ~8문항(필수)** + 선택 보강. 한 화면에 1~2문항씩 위저드.
2. **컨셉은 1회 잠금** — 설문 결과로 컨셉(페르소나·톤·기둥·비주얼)을 확정([SPEC 3.1](Kup_SPEC.md)). 이후 모든 생성의 제약으로 상속.
3. **하이브리드 입력** — 사용자가 씨앗을 주면(주제·기둥·톤) AI가 다듬어 제시(O2) → 사용자 확정. 빈손에서 만들지 않음.
4. **원본 보존** — 설문 raw를 `channel_configs.survey_raw`에 저장 → 컨셉 재도출·디버그용.

---

## 1. 설문 구조 (5섹션 · 위저드)

> 입력형식: `선택`(단일) · `다중`(복수) · `태그`(자유 입력 칩) · `주관식`(짧은 문장) · `슬라이더` · `컬러`
> ⭐ = 컨셉 도출 핵심 · 🔒필수 / ⚪선택

### 섹션 1 — 나/계정 (페르소나 토대)
| Q | 질문 | 입력형식 | 필수 |
|---|---|---|---|
| Q1 ⭐ | 어떤 분야로 계정을 운영하나요? | 선택(카테고리) + 기타 주관식 | 🔒 |
| Q2 ⭐ | 당신(계정)을 한 문장으로 소개한다면? | 주관식 | 🔒 |
| Q3 | 계정의 1차 목표는? (1K 팔로워 / 브랜딩 / 판매 유입) | 선택 | ⚪ |

### 섹션 2 — 콘텐츠 방향 (기둥·타깃)
| Q | 질문 | 입력형식 | 필수 |
|---|---|---|---|
| Q4 ⭐ | 주로 다룰 세부 주제 3~5개 | 태그(3~5) | 🔒 |
| Q5 ⭐ | 주 독자는 누구인가요? (연령·상황) | 주관식 또는 선택 | 🔒 |
| Q6 ⭐ | 그 독자가 겪는 고민·결핍은? (무엇을 모르거나 필요로 하나) | 주관식 | 🔒 |
| Q7 | 당신이 줄 수 있는 가치는? | 주관식 | ⚪ |

### 섹션 3 — 톤 & 금지선
| Q | 질문 | 입력형식 | 필수 |
|---|---|---|---|
| Q8 ⭐ | 말투/톤 분위기 | 다중(친근·전문·감성·유머·담백…) | 🔒 |
| Q9 ⭐ | **절대 쓰고 싶지 않은 표현/피할 것** | 태그 + 주관식 | 🔒 |
| Q10 | 이모지·유행어 사용 정도 | 슬라이더(적게↔많이) | ⚪ |

### 섹션 4 — 운영 (발행 리듬)
| Q | 질문 | 입력형식 | 필수 |
|---|---|---|---|
| Q11 ⭐ | 발행 빈도 목표 | 선택(주1·주2·주3·매일) | 🔒 |
| Q12 | 선호 발행 요일/시간대 | 다중 | ⚪ |

### 섹션 5 — 비주얼 (브랜드)
| Q | 질문 | 입력형식 | 필수 |
|---|---|---|---|
| Q13 ⭐ | 브랜드 컬러 | 컬러 + 프리셋 팔레트 | 🔒 |
| Q14 | 분위기 키워드(폰트/무드) | 선택 | ⚪ |

> 💡 비주얼 대안: Q1(분야)·Q8(톤) 기반으로 **팔레트를 AI가 먼저 추천** → 사용자가 고르거나 직접 조정. 빈 컬러피커보다 마찰↓.

## 2. 답 → `channel_configs` 매핑 (1.3)

| 컨셉 필드 | 주 출처 | 도출 방식 |
|---|---|---|
| `persona` | Q1·Q2·Q3 | "분야 + 자기소개 + 목표"를 1문장 페르소나로 AI 합성 |
| `tone` | Q8·Q10 | 선택 톤 + 이모지 강도 → 1문장 톤 지시문 |
| `pillars` (3~5) | Q4 (+ Q1) | 사용자 태그를 AI가 정제·중복제거 후 3~5개로 확정 |
| `cadence` | Q11 | 그대로 매핑 |
| `visual` | Q13·Q14 | 컬러 → primary/primary2/accent/light, 무드 → font |
| **금지표현**(시스템 프롬프트 불변규칙) | Q9 | [프롬프트 §1 불변규칙3](../tech/Kup_프롬프트_설계.md)에 주입 |
| **추천 주제 seed**(O2) | Q4·Q5·Q6 | "기둥 × 타깃 결핍" 교차로 ① 주제 제안에 사용 |
| `survey_raw` | 전체 | 원본 JSON 보존 |

> 도출 결과 = [파이프라인 §1.1 Concept](../tech/Kup_생성파이프라인_설계.md) 계약과 동일 구조 → 시스템 프롬프트가 바로 소비.

## 3. 설문 → 컨셉 → 추천 주제 흐름 (O1 → O2)

```
[O1 설문 제출]
   │  답 → survey_raw 저장
   ▼
[AI 컨셉 합성]  persona/tone/pillars/cadence/visual + 금지표현
   ▼
[O2 컨셉 카드 제시]  "이렇게 이해했어요" — 페르소나/톤/기둥/팔레트 미리보기
   │  ├─ [수정] → 해당 항목만 손보고 재합성
   │  └─ [이대로 확정] → channel_configs 잠금(locked_at)
   ▼
[추천 주제 N개]  기둥 × 타깃 결핍 교차 → ① 주제 제안 → 첫 카드뉴스로 연결
```

- **O2는 "확인·수정"이지 재입력이 아님** — 마찰 최소화.
- 확정 시 `locked_at` 기록. 이후 변경은 별도 "컨셉 수정" 경로(추후).
