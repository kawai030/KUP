## 1. 앱 리뷰(App Review) 상세

**왜 존재하나**: 우리 앱이 "남의 인스타 계정에 글을 올리고 DM을 보내는" 강력한 권한을 갖기 때문에, Meta가 "이 회사 진짜인지, 권한을 악용 안 하는지" 사전 심사한다.

**① 비즈니스 인증(Business Verification)이란**
- Meta가 "이 앱을 만든 주체가 실존하는 사업자인가"를 확인하는 절차.
- 제출물: 사업체 이름·주소·이메일·전화, 보통 **사업자등록증** 같은 공식 서류, 회사 도메인/웹사이트.
- ⚠️ **1인/초기 팀에 현실적 장벽**. 개인이 아니라 "사업자"로 인증해야 콘텐츠 발행·메시지 권한을 받기 쉽다. → **사업자등록(개인사업자라도)이 사실상 필요**할 가능성이 큼. 출시 전 미리 준비할 항목.

**② 권한별 화면녹화(Screencast)란**
- 권한 4종(발행·댓글·메시지·기본정보)을 신청할 때, **각 권한마다** "우리 앱에서 그 권한이 실제로 어떻게 쓰이는지"를 화면 녹화한 영상을 제출.
- 예) 발행 권한 → "사용자가 인스타로 로그인 → 시안 승인 → 게시물이 실제로 올라가는" 흐름을 영상으로.
- 심사관이 **우리가 제공한 테스트 계정으로 영상 속 단계를 똑같이 재현**할 수 있어야 통과. (반려 1순위 사유 = "녹화 불충분")

**③ 미리 준비할 체크리스트**
- 작동하는 앱(로그인 포함), 개인정보처리방침 URL, 앱 아이콘, 권한별 사용목적 설명문, 테스트 계정, 권한별 녹화영상, 사업자 인증서류.
- 소요 **2~4주**, 반려 흔함 → **기능이 돌아가는 시점에 바로 제출**해야 출시가 안 밀림.

**④ 개발 모드 vs 라이브 모드 (핵심 안심 포인트)**
- **개발 모드**: 앱에 역할(관리자/테스터)을 준 내 계정에만 기능 사용 가능 → **심사 없이 지금 테스트 OK** (= 테스트 3).
- **라이브 모드**: 실제 고객(아무 사용자)에게 서비스하려면 → 심사 통과 필수.
- 즉 **개발/PoC는 지금, 심사는 병렬로 준비.**

## 2. n8n이 무엇인가

**n8n = "코딩 없이 자동화 흐름을 블록으로 그려서 실행하는 도구".**
- 화면에서 노드(블록)를 선으로 연결: `[웹훅 들어오면] → [조건 확인] → [인스타 API 호출] → [슬랙 알림]`. 순서도를 그리면 그게 실제로 동작.
- 앱/서비스 수백 개 커넥터 제공, 직접 서버에 설치(셀프호스팅)하거나 클라우드 버전 사용.
- 강점: **비개발자도 빠르게** 자동화 가능. 약점: 복잡/민감한 로직엔 부적합, 별도 시스템을 또 운영해야 함.

**우리가 n8n을 쓴다면 어디에?**
- **① 스파이크(빠른 검증) 도구**: 코드 짜기 전에 n8n으로 "인스타 발행 흐름"을 블록 몇 개로 빠르게 성공시켜 보고 자신감 확보 → 이후 코드로 옮김.
- **② 내부 운영 자동화**: "발행 실패하면 슬랙 알림", "매주 지표 메일 발송" 같은 **핵심이 아닌 잡일**.
- 결론은 5번 참조.

## 3. 페이스북 페이지, 정말 필요 없나? (재확인 결과)

**정확히 두 가지 방식이 공존한다.** (Meta 공식 기준)

| | **Instagram API with Instagram Login** (신·권장) | **Instagram API with Facebook Login** (구·Graph API) |
|---|---|---|
| 로그인 | 인스타로 **바로** OAuth | 페이스북으로 로그인 |
| **페이스북 페이지** | **필요 없음** ✅ | **필요함** (계정↔페이지 연결 필수) |
| 적합 | 단일 계정(개인·소상공인이 자기 인스타 연결) | 여러 계정 중앙관리·대행사·비즈니스 매니저 통합 |
| 발행/댓글/DM | 지원 | 지원 |
| 계정 조건 | 둘 다 프로페셔널(비즈니스/크리에이터)만 |

- "페이스북 페이지 경유 Graph API"는 **여전히 존재**한다. 다만 2024년부터 Meta가 **페이지 없이 인스타로 바로 로그인하는 방식**을 추가했고, 우리 서비스(사용자가 자기 인스타 1개 연결)엔 이게 더 간단해서 권장.
- 둘 다 발행·댓글·DM을 지원 → 기능 손해 없음.
- 참고: 사용자가 이미 페이지 연결된 비즈니스 계정이면 구 방식도 가능 → 실제로는 **두 방식 둘 다 지원**해 온보딩 이탈을 줄이는 설계도 가능(나중 결정).

## 4. 인스타 계정 종류 정리

```
인스타 계정
├─ 개인(Personal) 계정 ........ API 사용 불가 ❌
└─ 프로페셔널(Professional) 계정 ... API 사용 가능 ✅
    ├─ 크리에이터(Creator) .... 인플루언서·창작자용 (성장지표 등)
    └─ 비즈니스(Business) ...... 상점·브랜드용 (연락처·카탈로그 등)
```

- 전환은 **무료·즉시·되돌리기 가능**(앱 설정).
- 우리 서비스는 **비즈니스/크리에이터 둘 다** 수용. (둘 다 발행·댓글·DM API 가능. 세부 인사이트 지표만 약간 차이)
- 온보딩에서 "개인 계정이면 → 프로페셔널 전환 안내" 화면 필요(SPEC 3.1 실패 시나리오와 일치).

## 5. 백그라운드 작업/예약 + n8n 필요한가 (상세)

**용어**
- **백그라운드 작업** = 사용자가 화면을 안 봐도, 정해진 시각에 알아서 실행돼야 하는 일. 예: "내일 아침 9시에 이 게시물 발행".
- **cron(크론)** = "매시간/매일 N시 실행" 같은 **시간표 스케줄러**. (예: 1분마다 "지금 발행할 예약 건 있나?" 확인)
- **BullMQ** = Node.js용 **작업 대기열(큐) 라이브러리**, Redis(빠른 메모리 저장소) 사용. "이 작업을 9시에 실행", "실패하면 3번 재시도" 같은 **지연·재시도·순서**를 안정 처리.
- 둘이 합쳐 = "예약 발행, 실패 재시도, 웹훅 처리"를 우리 코드 안에서 직접 담당.

**"n8n 대체"의 뜻 = n8n 없어도 된다는 말.**
- 즉 **프로덕션(실서비스)에는 n8n을 넣지 않는다.** 발행·예약·재시도·웹훅이라는 핵심 로직을 BullMQ+cron으로 **우리 코드에 둔다.**
- 이유: 핵심 로직은 코드로 둬야 (1) 버전관리·테스트 (2) 에러추적 (3) 인스타 토큰 보안 (4) 세밀한 재시도 제어가 된다. n8n을 핵심에 넣으면 별도 시스템을 또 운영·보안해야 하고 토큰이 한 군데 더 지나가 위험이 커진다.
- **n8n은 "있어도 되고 없어도 되는" 보조**: ① 초기 검증 스파이크 ② 내부 잡일 자동화. **MVP에선 안 써도 전혀 문제없음**(권장: 미사용).

**한 줄 정리**: 예약/발행/재시도 = **우리 백엔드 코드(BullMQ+cron)**. n8n = 선택(권장: MVP 미사용).

## 6. AI 모델, Claude vs Gemini — 테스트로 결정

품질·비용 둘 다 **우리 실제 프롬프트로 비교 테스트**해야 정답이 나온다.

단가 비교(100만 토큰당, 2026.6):

| 모델 | 입력 | 출력 | 비고 |
|---|---|---|---|
| Gemini 3.5 Flash | $1.50 | $9 | 저렴·빠름 |
| Gemini 3.1 Pro | $2 | $18 | |
| Claude Haiku 4.5 | $1 | $5 | 가장 저렴 |
| Claude Sonnet 4.6 | $3 | $15 | 품질·비용 균형 |
| Claude Opus 4.8 | $5 | $25 | 최고 품질 |

- 둘 다 **배치 API 50% 할인** 있음(급하지 않은 대량 생성에 유리).
- 일반적으로 Gemini Flash가 더 싸고, Claude는 지시 따르기·한국어 카피 뉘앙스·자가점검에서 강한 편 — **단, 우리 주제(재테크/패션 등)·톤으로 직접 블라인드 비교해야** 확정.
- **설계 권장**: 생성 단계를 **"모델 교체 가능"하게**(어댑터 패턴) 만들어, 같은 프롬프트를 Claude Sonnet vs Gemini Flash에 돌려 ① 결과물 품질 블라인드 채점 ② 건당 비용·속도 측정. 테스트 1 때 같이 설계.

## 7. 서버 올리기(배포/호스팅)

"서버를 띄운다" = **우리 코드가 24시간 켜져 있는 컴퓨터에서 돌아가야** 하고, 인터넷 누구나 접속 가능해야 함. 올리는 방법(난이도별):

| 방식 | 예시 | 난이도 | 우리에게 |
|---|---|---|---|
| **PaaS** (관리형, 코드만 올리면 끝) | **Railway, Render, Fly.io**(백엔드·워커) + **Vercel**(프론트) | 쉬움 | ✅ **3인 팀 권장** |
| 관리형 백엔드 | **Supabase**(DB·인증·스토리지) | 쉬움 | ✅ 같이 사용 |
| **IaaS** (직접 서버 관리) | **AWS EC2**, GCP | 어려움 | ❌ MVP엔 과함 |
| AWS 상위 서비스 | AWS App Runner/Amplify | 중간 | 나중 선택지 |

- **권장 조합**: 프론트=Vercel, 백엔드/워커=Railway 또는 Render, DB·인증·스토리지=Supabase.
- "git에 코드 올리면(push) 자동 배포"되는 방식이라 서버 관리 부담이 적음. **AWS 직접 운영은 초기엔 비추**. 나중에 규모 커지면 이전 가능.
- 비용: 초기엔 **월 수십 달러 수준**으로 시작 가능.
- ⚠️ 카드 렌더(Playwright)와 작업큐는 "상시 켜진 워커 서버"가 필요 → 프론트(Vercel)만으론 안 되고 Railway/Render 같은 워커 호스팅이 한 개 더 필요.

## 8. 모니터링

"서비스가 잘 돌아가는지 감시"하는 장치들:

| 종류 | 무엇 | 도구 예시 |
|---|---|---|
| 에러 추적 | 코드가 터지면 알림+원인 | **Sentry** |
| 로그 | 무슨 일이 있었는지 기록 검색 | Logtail, Axiom |
| 가동 감시(Uptime) | 사이트가 죽으면 알림 | UptimeRobot, BetterStack |
| 제품 분석 | 가입·발행·리드 퍼널, KPI | PostHog, Amplitude |

- **우리에게 가장 중요한 것**: "예약된 발행이 실제로 성공했나 / 웹훅(댓글)이 누락 없이 왔나" 추적. → 발행 잡 성공·실패를 DB에 기록하고, **실패하면 슬랙으로 즉시 알림** + 간단한 내부 대시보드. (KPI 가드레일 "고위험 검수 누락률 0%"와 연결.)
- MVP 시작: **Sentry(무료티어) + UptimeRobot + 자체 발행로그 + 슬랙알림**이면 충분.

## 9. 카드뉴스 이미지를 어떻게 만드나

**"이미지를 만든다"에 두 가지가 있고, 우리가 쓰는 건 2번.**

**① AI 이미지 생성 (text-to-image)** — DALL·E, Midjourney, Imagen, Stable Diffusion 류
- 프롬프트로 사진/일러스트를 통째로 그려냄.
- 카드뉴스엔 **부적합**: 이미지 안의 글자가 깨지고, 손가락 등 아티팩트, 브랜드 톤 일관성 깨짐, 비쌈, 통제 어려움 → 고위험 검수 폭증 → "신뢰" 포지션과 충돌.

**② 템플릿 + 텍스트 배치 (✅ 우리 방식)** — PoC가 한 그것
- 우리가(또는 AI가 한 번) **카드 템플릿(레이아웃·브랜드 컬러·폰트)을 디자인**해 둠.
- AI는 **글(카피)만** 생성.
- 코드가 그 텍스트를 템플릿의 정해진 자리에 **배치 → PNG로 렌더**. (= "첫 월급" 카드 PoC)
- 결과: 항상 깔끔·온브랜드, 저렴, 검수가 진짜 5초.

**정리**: **"템플릿(우리가 디자인) + AI가 만든 텍스트를 적절히 배치"** 형식이 맞다. AI는 "템플릿을 처음 몇 개 만들 때" 도움을 줄 수도 있고(1회성), 매 게시물에선 텍스트만 채운다.
- 사진이 꼭 필요한 경우만(B 모드 제품사진) 사용자 업로드 사용.
- AI 생성 이미지는 "배경 장식"(글자 없는 영역) 정도로만 선택적 사용 → 후순위.

## 10. "Claude Design"이란? 연결 가능?

2026년 4월 17일 Anthropic이 **"Claude Design"** 출시. Opus 4.7 기반, **Canva와 통합**, 디자인·프로토타입·슬라이드를 대화로 제작. 코드베이스/디자인파일을 읽어 **디자인 시스템(컬러·타이포·컴포넌트)을 자동 구성**. 현재 Claude Pro/Max/Team/Enterprise 대상 리서치 프리뷰.

**우리 서비스에 "연결"되나?**
- **직접 API로 끼워넣는 건 안 됨.** Claude Design은 **사람이 쓰는 챗 제품**이지, 우리 앱이 호출하는 API가 아님.
- 활용법 2가지:
  1. **팀이 직접 사용** → 우리 카드 **마스터 템플릿을 빠르게 디자인**하는 보조 도구로.
  2. 밑단 엔진인 **Canva는 API(Connect API) 공개** → 프로그램으로 디자인 자동화하려면 **Canva Connect API** 사용(11번).

## 11. Canva / Kling / Higgsfield 연결

전부 **API로 연결 가능**. 용도가 다름:

**① Canva Connect API** — **카드 이미지 제작의 대안**
- OAuth 연결 후, Canva에 만든 **브랜드 템플릿에 데이터(텍스트)를 자동 주입(autofill) → PNG 내보내기**.
- 9번의 "HTML/CSS 직접 렌더" 대신 쓸 수 있는 선택지. **장점**: 예쁘고 온브랜드·쉬움. **단점**: Canva 구독 비용·외부 의존·픽셀 통제 약함.
- → **테스트 가치 충분.** "HTML/CSS 렌더 vs Canva API" 비교 후 결정.

**② Kling AI** — **릴스/숏츠 영상 생성**
- 텍스트/이미지 → 영상. 증강 모드 C·릴스 보조. 건당 과금.

**③ Higgsfield** — **영상/이미지 생성 인프라**
- Kling·Veo 등 30+ 모델을 한 곳에서. 영상 레이어, **후순위(M2)**.

**정리**
- **Canva** → 카드뉴스 이미지 **대안 경로** (지금 검토 가치 O).
- **Kling / Higgsfield** → **릴스 영상** 레이어 (보조·M2, MVP 막판/이후).

## 핵심 결정/정정 사항 요약

- 생성 단계는 **모델 교체 가능(어댑터)** 설계 → Claude vs Gemini 블라인드 비교 후 결정.
- 카드 이미지: **템플릿+텍스트 렌더** 기본. **Canva Connect API**를 대안으로 비교 검토.
