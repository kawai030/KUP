# Kup — 개발/기술 용어집 (계속 업데이트)

> 비개발자 기획자가 이해하기 위한 용어 사전. 모르는 용어가 나오면 여기에 계속 추가.
> 설명 원칙: 쉬운 비유 + 한 줄 정의 + 우리 프로젝트에서의 쓰임.
> 자주 쓰는 비유 = **식당 차리기**: 코드=요리사·메뉴 / 서버=건물 / 호스팅=임대 / 배포=입주·개업.

---

## 🏗️ 인프라 · 배포 (서비스를 띄우는 것)

### 서버 (Server)
- **정의**: 내 코드가 24시간 돌아가는 컴퓨터(장소).
- **비유**: 식당 **건물** 자체.
- **포인트**: 내 노트북은 끄면 멈추고 남이 못 찾아옴 → 항상 켜진 컴퓨터가 필요. "어디서 돌아가나"에 답하는 것.

### 호스팅 (Hosting)
- **정의**: 그 서버(컴퓨터)를 빌리는 것/서비스.
- **비유**: 건물 **임대 계약**.
- **쓰임**: Vercel·Railway·AWS가 호스팅 업체(임대인). 매달 임대료 내고 공간을 빌림.

### 배포 (Deploy / Deployment)
- **정의**: 내가 만든 코드를 서버에 올려서 **실제로 돌아가게 만드는 행위**.
- **비유**: 건물에 **입주해서 재료 채우고 영업 시작**.
- **포인트**: 코드를 고칠 때마다 다시 배포(다시 차려놓기). 요즘은 git에 push하면 자동 배포되는 경우가 많음.

### 로컬 개발 서버 (localhost · `npm run dev`)
- **정의**: 내 컴퓨터에서만 앱을 띄워 직접 확인하는 것. `localhost`(=내 컴퓨터 자신의 주소, 나만 보임)로 접속.
- **비유**: 개업(배포) 전 **주방 시운전** — 손님 받기 전 직접 한 그릇 만들어봄.
- **쓰임**: `npm run dev` → `http://localhost:3000`에 화면이 뜨면 뼈대 정상. **배포(공개)와 반대 개념**(localhost는 인터넷에 공개 안 됨).

### PaaS (Platform as a Service)
- **정의**: "코드만 올리면 알아서 돌려주는" 관리형 호스팅. 서버 관리(전기·보안·확장)를 업체가 대신 해줌.
- **비유**: **풀옵션 임대 매장** (인테리어·설비 다 돼 있어 들어가서 장사만).
- **예시**: Vercel(프론트), Railway·Render·Fly.io(백엔드·워커). → **3인 팀 권장**.

### IaaS (Infrastructure as a Service)
- **정의**: 빈 서버(컴퓨터)만 빌려주고, 설정·관리는 내가 다 하는 방식.
- **비유**: **빈 건물만 임대** (인테리어·설비 직접).
- **예시**: AWS EC2, GCP. 자유롭지만 할 일 많음 → MVP엔 과함, 나중에 경험/규모 커질 때.

### 관리형 백엔드 (Managed Backend) / Supabase
- **정의**: DB·로그인·파일저장 같은 "데이터 관련 기능"을 한 번에 제공하는 관리형 서비스. *코드를 돌리는 서버가 아니라 데이터 서비스.*
- **비유**: 식당의 **창고+금고+예약대장**을 대신 관리해주는 업체.
- **예시**: Supabase(PostgreSQL DB + 회원 인증 + 파일 스토리지).

### Vercel / Railway / Render
- **Vercel**: 프론트엔드(사용자 화면) 호스팅에 강함.
- **Railway / Render**: 백엔드·상시 워커 호스팅. (Railway는 DB·Redis도 같이 제공 가능)
- **포인트**: "화면 둘 곳 / 코드 돌릴 곳 / 데이터 둘 곳" 세 역할 — 한 곳에 몰 수도, 나눌 수도 있음(3개 의무 아님).

### AWS (Amazon Web Services)
- **정의**: 세계 최대 클라우드. 서버·DB·스토리지 등 수백 가지 서비스. 많은 PaaS도 내부적으로 AWS 위에서 돎.
- **포인트**: MVP엔 직접 운영 비추(할 일 많음). 초보 친화 서비스(Amplify·App Runner·Lightsail·RDS)도 있음. **MVP 후 경험삼아 띄워보면 학습 가치 큼.**

## ⚙️ 백그라운드 작업 · 자동화

### cron (크론)
- **정의**: "매시간/매일 N시 실행" 같은 **시간표 스케줄러**.
- **쓰임**: 1분마다 "지금 발행할 예약 건 있나?" 자동 확인.

### BullMQ
- **정의**: Node.js용 **작업 대기열(큐)** 라이브러리. "이 작업 9시에 실행", "실패하면 재시도" 같은 지연·재시도·순서를 안정 처리.
- **쓰임**: 예약 발행, 실패 재시도, 웹훅 처리. → n8n 없이 이걸로 핵심 자동화.

### 작업 큐 (Job Queue)
- **정의**: 해야 할 일(잡)을 줄 세워두고 워커가 하나씩 처리하는 구조.
- **비유**: 식당 주방의 **주문표 꽂이** — 들어온 순서대로 처리, 실패하면 다시.

### 워커 (Worker)
- **정의**: 큐에 쌓인 작업을 실제로 실행하는 "상시 일하는 프로그램".
- **비유**: 주문표를 보고 요리하는 **주방 직원**. 항상 켜져 있어야 함.

### Redis (레디스)
- **정의**: 매우 빠른 메모리 기반 저장소. BullMQ가 작업 큐를 보관하는 데 사용.
- **비유**: 주방 옆 **포스트잇 보드** (빠르게 붙였다 뗐다).
- **쓰임**: 관리형 Redis(Upstash 등)로 간단히 추가.

### n8n
- **정의**: 코딩 없이 자동화 흐름을 **블록 그림**으로 그려 실행하는 도구.
- **비유**: 레시피를 **카드 드래그**로 만드는 것 (코드는 글로 적는 것).
- **우리 입장**: 핵심 로직은 코드(BullMQ)로. n8n은 ① 빠른 검증 연습장 ② 내부 잡일 자동화에만 선택적. **MVP 미사용 권장.**

## 🔌 API · 연동

### API (Application Programming Interface)
- **정의**: 프로그램끼리 대화하는 "정해진 창구/메뉴판". 우리가 정해진 형식으로 요청하면 정해진 형식으로 응답.
- **비유**: 식당 **주문 창구** — 메뉴판대로 시키면 음식이 나옴(내부 주방은 몰라도 됨).
- **예시**: 인스타 API(발행·DM), Claude API(텍스트 생성), Canva API(디자인).

### OAuth (오어스)
- **정의**: 비밀번호를 넘기지 않고 "이 앱이 내 계정의 이 기능을 쓰도록 허락"하는 안전한 위임 로그인 방식.
- **비유**: 집 열쇠를 통째로 주는 대신, **특정 문만 열리는 임시 출입증**을 발급.
- **쓰임**: 사용자가 인스타 계정을 우리 서비스에 연결할 때. (비번 미보관, 토큰만 받음)

### 토큰 (Access Token)
- **정의**: OAuth로 받은 "출입증". 이걸로 그 계정에 발행·조회 가능. 만료 기한 있음(인스타 장기토큰 60일).
- **포인트**: 절대 공개 금지, 암호화 저장, 만료 전 갱신 필요.

### 인스타 OAuth ("Instagram API with Instagram Login")
- **정의**: 위 OAuth를 인스타에 적용한 우리 실제 연결 방식. 2024년 이후 Meta가 미는 방식으로, **페이스북 페이지 연결 없이** 인스타로 바로 로그인해 그 계정의 토큰을 받는다.
- **비유**: 손님(사용자)이 집 열쇠(비번)를 주는 대신, 경비실(Meta)이 발급한 **임시 출입증(토큰)**만 우리 가게에 맡기는 것. 출입증으로는 정해진 일(발행·댓글·DM)만 가능, 60일 후 만료.
- **연결 흐름**: ① "인스타로 로그인" 버튼 → ② Meta 동의화면(권한 승인) → ③ 우리 백엔드가 인증 코드 수신 → ④ 단기 토큰 받기 → ⑤ **장기 토큰(60일)으로 교환** → ⑥ **암호화해 DB 저장**(`ig_tokens`). **비번은 절대 안 받음.**
- **전제**: 사용자 인스타가 **프로페셔널 계정(비즈니스/크리에이터)**이어야 함. 개인 계정은 API 불가(전환 30초·무료).
- **헷갈림 주의**: '우리 서비스 회원 로그인'(Supabase Auth)과 '인스타 OAuth'(계정 연결)는 **별개의 두 단계**.
- **비용 $0**(Meta API 호출은 무료). 단, **남의 계정에 쓰려면 Meta 앱 리뷰**(2~4주) 통과 필요. 개발/테스트는 앱 리뷰 없이 가능(내 계정·테스터 한정).

### 웹훅 (Webhook)
- **정의**: "사건이 생기면 그쪽에서 우리 서버로 알려주는" 역방향 알림. (우리가 계속 물어보지 않아도 됨)
- **비유**: 택배 도착 시 **자동으로 오는 문자 알림**.
- **쓰임**: 인스타에 댓글이 달리면 Meta가 우리 서버로 이벤트 전송 → 우리가 DM 발송.

### 프론트엔드 / 백엔드
- **프론트엔드(Frontend)**: 사용자가 보는 화면(웹/앱 UI). 식당의 **홀·메뉴판**.
- **백엔드(Backend)**: 화면 뒤에서 도는 처리·데이터·로직. 식당의 **주방**.

## 🤖 AI · 콘텐츠 생성

### 토큰(Token) — AI 맥락
- **정의**: AI가 글을 처리하는 단위(대략 단어 조각). API 비용이 토큰 수로 계산됨.
- **포인트**: 출력 토큰이 입력보다 비쌈(보통 5배) → 카피는 짧고 군더더기 없이.

### text-to-image (텍스트→이미지 생성)
- **정의**: 프롬프트로 사진/그림을 통째로 그려내는 AI. (DALL·E, Midjourney, Imagen 등)
- **우리 입장**: 카드뉴스엔 부적합(글자 깨짐·아티팩트). **템플릿+텍스트 배치 방식**을 씀.

### 이미지 캐러셀 (Carousel)
- **정의**: 사진 여러 장(최대 10장)을 한 게시물에 담아 옆으로 넘겨보는 인스타 게시 형식. 사진 아래 점(●○○)이 여러 개인 그 게시물.
- **비유**: 한 접시(게시물 1건)에 여러 요리를 **코스처럼** 담아 한 번에 내는 것 — 손님은 하나씩 넘겨가며 봄.
- **우리 쓰임**: **카드뉴스 = 캐러셀**. 여러 장의 카드 슬라이드를 캐러셀 1건으로 발행.
- **포인트**: **캐러셀 1개 = 발행 1건**으로 카운트 → 슬라이드 10장이어도 발행 한도(24시간 약 25건)는 1건만 소모. 카드뉴스 전략에 유리.
- **발행 방식**: 3단계 컨테이너 발행(① 컨테이너 생성 → ② 처리완료 FINISHED 확인 → ③ 게시). PoC 테스트3에서 실제 발행 검증됨.

### 템플릿 렌더링 (Template Rendering)
- **정의**: 미리 만든 디자인 틀(템플릿)에 텍스트를 넣어 이미지(PNG)로 출력.
- **우리 방식**: HTML/CSS 템플릿 → Playwright로 PNG 변환. (PoC로 검증됨)

### Playwright (플레이라이트)
- **정의**: 화면이 안 보이는 브라우저를 코드로 조종하는 도구. 웹페이지를 캡처해 이미지로 만들 수 있음.
- **쓰임**: 카드 템플릿(HTML)을 PNG로 렌더.

### 프롬프트 (Prompt)
- **정의**: AI에게 주는 지시문/입력.
- **쓰임**: "이 컨셉·톤으로 카드뉴스 슬라이드 카피 써줘" 같은 명령.

### 구조화 출력 (Structured Output) / JSON 스키마
- **정의**: AI에게 자유 줄글 대신 **정해진 칸(필드)에 맞춰서만** 답하게 형식을 강제하는 것.
- **비유**: 빈 종이 대신 **정해진 주문서 양식**을 주고 칸만 채우게 하는 것.
- **우리 쓰임**: 카드뉴스 deck(슬라이드·캡션·해시태그)을 항상 같은 JSON 구조로 받아 렌더러에 바로 투입.

### Zod (조드) — 스키마 검증
- **정의**: 들어온 데이터가 "정해진 형식·규칙"에 맞는지 코드로 자동 검사하는 도구(TypeScript용).
- **비유**: 주방에 들어온 **재료가 발주서 규격(개수·크기·신선도)에 맞는지 확인하는 검수 담당**. 안 맞으면 "반품!".
- **우리 쓰임**: AI가 만든 deck JSON이 ① 필드 다 있는지 ② **글자수 제한 안 넘었는지** 검사 → 통과해야 저장·렌더. 실패 시 1회 재생성(repair).
- **포인트**: AI는 가끔 형식을 어김 → Zod가 **문지기**. "AI 출력을 믿지 말고 검사한다."

## 🛡️ 모니터링 · 운영

### 모니터링 (Monitoring)
- **정의**: 서비스가 잘 도는지 감시·기록하는 것.
- **종류**: 에러추적(Sentry), 제품분석(PostHog), 가동감시(UptimeRobot), 발행로그.

### Sentry (센트리)
- **정의**: 코드가 터지면 위치·원인을 잡아 알려주는 에러 추적 도구.

### 슬랙 Incoming Webhook
- **정의**: 슬랙이 주는 URL로 메시지를 POST하면 채널에 자동으로 글이 뜨는 기능.
- **쓰임**: 발행 실패·신규 가입·첫 자동발행(Aha) 같은 이벤트 알림.

## 🌿 협업 · 버전관리 (동시 개발)

> 여러 명이 같은 코드를 동시에 고칠 때 안 부딪히게 하는 도구·개념. 실전 규칙은 [CONTRIBUTING.md](../../CONTRIBUTING.md) §1·§1.5.

### 브랜치 (Branch)
- **정의**: 코드의 "평행 우주" 복사본. 내 브랜치에서 뭘 하든 남의 브랜치엔 영향 없음.
- **비유**: 같은 요리책을 **각자 복사해서** 자기 버전으로 고치는 것. 나중에 좋은 것만 원본에 합침.
- **우리 쓰임**: 각자 `feature/<영역>-<요약>` 브랜치를 **`dev`에서 따서** 작업.

### main / dev / feature
- **main**: 배포되는 안정판. 개인은 직접 안 건드림.
- **dev**: 모두의 작업이 모이는 통합 브랜치. **여기가 실질적 최신**.
- **feature/***: 각자 작업하는 일회용 브랜치. dev에서 따고 dev로 PR.
- **비유**: feature=각자 주방 / dev=합동 시식대 / main=손님에게 내는 정식 메뉴.

### 커밋 (Commit)
- **정의**: 변경사항을 "이 시점으로 저장"하는 한 묶음. 메시지로 뭘 했는지 기록.
- **포인트**: 작게 자주. 한 커밋 = 한 가지 일.

### 스테이징 / `git add`
- **정의**: 커밋에 포함할 파일을 **고르는** 단계. add 한 것만 커밋됨.
- **주의**: `git add -A`(전부 담기)는 **남이 만들어둔 다른 변경까지 쓸어담을 수 있음** → 커밋 전 `git status`로 뭐가 담기는지 꼭 확인.

### push / fetch / pull
- **push**: 내 브랜치를 GitHub에 올리기(백업·공유). **합쳐지는 게 아님.**
- **fetch**: GitHub 최신 정보 받아오기(아직 내 코드엔 안 섞음).
- **pull**: fetch + 내 현재 브랜치에 합치기.

### merge (병합)
- **정의**: 두 브랜치를 합치는 것. 합친 자국(머지 커밋)을 남김.
- **비유**: 두 사람의 레시피 수정본을 **하나로 합쳐 적기**.
- **우리 쓰임**: ① PR이 dev에 들어갈 때 ② 작업 중 dev 최신을 내 브랜치로 끌어올 때. **우리 팀은 merge로 통일**(rebase 안 씀).

### rebase (리베이스)
- **정의**: merge와 같은 목적(브랜치 합치기)인데, 내 커밋을 상대 브랜치 끝에 **다시 쌓아** 히스토리를 일직선으로 만드는 방식.
- **왜 우리는 안 쓰나**: 커밋이 다시 써져서 `--force` push가 필요 → 초보가 쓰면 작업이 날아갈 위험. **깔끔함보다 안전 우선** → 우리는 merge만.

### 충돌 (Conflict)
- **정의**: 두 사람이 **같은 줄**을 다르게 고쳐서 git이 누구 걸 쓸지 못 정하는 상태.
- **비유**: 레시피 같은 줄에 두 사람이 다른 메모 → "둘 중 뭐가 맞아?" 표시(`<<<<<<<`).
- **해결**: 고장 아님. 마커 보고 올바른 쪽으로 직접 정리 → `git add` → 커밋. (CONTRIBUTING §1.5)

### 드리프트 (Drift)
- **정의**: 내 브랜치가 dev에서 오래 안 합쳐져 점점 벌어진 상태. 동시 개발 충돌의 **진짜 원인**.
- **예방**: 작업 중 **하루 한 번 dev를 내 브랜치로 끌어오기**(`git merge origin/dev`). 매일 조금씩 합치면 안 아픔.

### PR (Pull Request)
- **정의**: "내 브랜치를 dev에 합쳐주세요" 요청. 팀원이 코드 보고 승인하면 머지.
- **비유**: 내 레시피 수정본을 **합본에 넣어달라고 검토 요청** 올리는 것.
- **우리 규칙**: 무엇·왜·테스트법 적기 / 리뷰어 1명 승인 / 셀프머지 금지 / 3종 검사 통과(CONTRIBUTING §3).

### worktree (워크트리)
- **정의**: 같은 저장소를 **여러 폴더에 동시에 체크아웃**하는 git 기능. 폴더마다 다른 브랜치를 둘 수 있음.
- **비유**: 같은 요리책을 **책상 두 개에 펼쳐놓고** 한쪽은 A요리, 한쪽은 B요리 작업.
- **우리 쓰임**: 한 작업 디렉토리에서 여러 작업이 섞이지 않게 분리(머신당 1회 add — [작업트랙.md](../작업트랙.md)).

## 🧩 기타

### git / push
- **git**: 코드 변경 이력을 관리하는 도구(누가·언제·뭘 바꿨나).
- **push**: 내 코드를 공용 저장소(GitHub 등)에 올리는 행위. 이때 자동 배포가 연결되기도 함.

### PoC (Proof of Concept, 개념 증명)
- **정의**: "이게 진짜 되나?"를 작게 빨리 검증하는 시제품/테스트.
- **예시**: 우리가 만든 카드 렌더 테스트, 인스타 발행 테스트.

### MVP (Minimum Viable Product, 최소기능제품)
- **정의**: 핵심 가치를 증명할 수 있는 **최소한의 기능만** 갖춘 첫 버전.
