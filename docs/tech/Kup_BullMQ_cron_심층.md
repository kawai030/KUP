## 0. 1분 결론 (먼저)

1. **구현 가능성: 높음(★★★★☆).** cron+BullMQ는 "예약·재시도·웹훅 처리"를 위한 **업계 표준·검증된 라이브러리**다. 새로 발명할 게 없고, 우리 발행 로직(2단계 컨테이너)은 이미 검증 PoC(비공개)에서 코드로 성공했다. 남은 건 그 로직을 "워커가 큐에서 꺼내 실행하는 형태"로 옮기는 것뿐. **새 기술 리스크라기보다 정형화된 이식 작업.**

2. **돈은 딱 두 군데서 난다**: ① **Redis(큐 보관소)** ② **상시 켜진 워커 서버**. 둘 합쳐 **MVP 기준 월 $15~25 수준**에서 시작.

3. **★ 반드시 알아야 할 함정 1개**: BullMQ는 **큐가 비어 있어도 Redis를 계속 들여다본다(폴링).** Upstash의 **"건당 과금(Pay-As-You-Go)"**을 쓰면 *일을 안 해도* 명령 수가 쌓여 요금이 샌다 → **Upstash는 "고정요금(Fixed) $10 플랜"**을 쓰거나, **워커 옆에 Redis를 직접 띄워라**(건당 과금 없음). Upstash 공식 문서가 이걸 명시적으로 경고한다.

4. **대안도 알아둘 것**: Redis가 부담되면, **이미 쓰는 Supabase Postgres 위에서 도는 큐(pg-boss)**로 Redis 자체를 안 쓰는 길도 있다(§5). MVP에선 진지한 선택지.

## 1. 무엇을 만드는가 — 3개의 등장인물

"오케스트레이션"이라는 말이 거창해 보이지만, 부품은 셋뿐이다. **식당 주방** 비유로 고정한다.

| 부품 | 정체 | 식당 비유 | 누가 만드나 |
|---|---|---|---|
| **cron** | "매시각/매일 X시에 한 번씩" 반복 알람 | 주방 **벽시계 타이머** ("매일 새벽 3시에 재고 확인해") | 라이브러리/호스팅 내장 기능 |
| **BullMQ** | 개별 "할 일"을 줄 세워 보관·예약·재시도·추적하는 **작업큐 라이브러리** | **주문 꽂이 + 진행관리 보드** | npm 라이브러리(우리가 잡 정의) |
| **워커(Worker)** | 큐에서 일을 하나씩 꺼내 **실제로 실행**하는 상시 프로그램 | **요리사** (계속 출근해 있음) | 우리가 작성(발행 로직) |

여기에 **Redis**(큐의 데이터를 실제로 담는 초고속 메모장)가 깔린다. BullMQ는 "요리사에게 주문 꽂이를 다루는 법을 가르치는 매뉴얼"이고, 그 주문 꽂이의 물리적 실체가 Redis다.

### cron과 BullMQ의 분업 (헷갈리지 말 것)
- **cron = 반복 알람**. "매일 8시 실행" 같은 *주기성*만 담당. 개별 작업의 재시도·진행추적은 못 한다.
- **BullMQ = 개별 작업의 예약·재시도·멱등·추적**. "이 시안 하나를 *정확히 이 시각*에, 실패하면 *3번 백오프 재시도*하고, *진행상태를 기록*."
- **우리는 둘 다 쓴다**: cron이 "지금 수집할 채널들"을 매일 깨워서 → BullMQ 큐에 잡을 **투입**하고 → 워커가 처리. (cron=깨우개, BullMQ=처리기)

### 우리가 큐에 올릴 4종류 잡
[기술설계 ⑥](Kup_기술설계.md) 기준:
1. **발행 잡** — 승인된 시안을 예약 시각에 인스타 2단계 발행 (가장 중요)
2. **재시도/알림** — 발행 실패 시 백오프 재시도, 계속 실패면 사용자 알림
3. **웹훅 처리 잡** — 댓글 이벤트 수신 → 키워드 매칭 → Private Reply DM (7일·1회)
4. **인사이트 수집 잡** — cron이 매일 채널별로 투입 → Insights 수집 → metrics 저장

## 2. 실제로 어떻게 도는가 (동작 원리 + 코드 골격)

비개발자도 "아, 이렇게 돌겠구나"가 그려지도록, 각 잡을 **글 + 최소 코드 골격**으로 보인다. (실제 코드는 더 다듬지만, 모양과 분량 감은 이게 거의 전부다 — 그만큼 단순하다.)

### 2.1 예약 발행 — 핵심 흐름

```
[검수 UI] 사용자가 승인 + "내일 8시 예약"
   │  → 웹앱(Vercel)이 BullMQ 큐에 "발행 잡"을 delay=내일8시 로 등록하고 즉시 응답 끝
   ▼
[Redis 큐]  잡이 8시까지 잠자며 대기  (서버가 재시작돼도 Redis에 남아있어 안 사라짐)
   ▼ (8시 도달, 워커가 깨움)
[발행 워커(상시 서버)]  잡을 꺼냄 →
   1) AI 라벨 부착·캡션 최종화
   2) 인스타 컨테이너 생성(슬라이드 이미지 URL들)
   3) FINISHED 될 때까지 상태 폴링
   4) publish 호출 → 게시물 ID 수신 → posts 테이블 저장
   실패 시 → 백오프 후 재시도(예: 3회), 끝내 실패면 사용자 알림 + 검수피드 표시
```

**(A) 웹앱 쪽 — 잡 등록 (한 줄짜리 일)**
```ts
// 검수 승인 핸들러 안. "예약만" 걸고 바로 끝낸다(발행을 직접 안 함).
await publishQueue.add(
  'publish-deck',
  { draftId, channelId },                 // 워커가 받을 재료
  {
    delay: msUntil(scheduledAt),          // 내일 8시까지의 밀리초
    attempts: 3,                          // 실패 시 최대 3회
    backoff: { type: 'exponential', delay: 60_000 }, // 1분→2분→4분
    jobId: `publish:${draftId}`,          // ★ 멱등키: 같은 시안 중복 등록 방지
  }
);
```

**(B) 워커 쪽 — 실제 발행 (PoC 로직 이식)**
```ts
const worker = new Worker('publish', async (job) => {
  const { draftId, channelId } = job.data;
  const token = await decryptToken(channelId);          // 금고에서 토큰 꺼냄
  const containerId = await ig.createCarousel(draftId, token);  // ① 컨테이너
  await ig.waitUntilFinished(containerId, token);       // ② FINISHED 폴링
  const mediaId = await ig.publish(containerId, token); // ③ 게시
  await db.posts.markPublished(draftId, mediaId);       // 결과 저장
}, { connection: redis, concurrency: 5 });

worker.on('failed', (job, err) => notifyUserAndAudit(job, err)); // 끝내 실패 시
```

> 핵심 체감: **"예약·재시도·추적"이라는 어려운 부분은 BullMQ가 다 해준다.** 우리가 쓰는 건 `add()`로 넣고, 워커 함수 안에 **이미 PoC에서 성공시킨 발행 로직을 그대로** 넣는 것. 새로 짜는 양이 적다.

### 2.2 재시도 · 백오프 · 멱등 (신뢰성의 핵심, BullMQ 기본 제공)
- **재시도/백오프**: 위 `attempts`·`backoff` 옵션만으로 "실패하면 1분→2분→4분 뒤 다시"가 자동. 직접 구현 불필요.
- **멱등(idempotency)**: "같은 잡이 두 번 실행돼 **중복 발행**되는 사고" 방지가 제일 중요. 두 겹으로 막는다:
  1. `jobId`로 **큐 등록 단계** 중복 차단(같은 draft는 한 번만 큐에 들어감).
  2. 워커 안에서 **발행 전 `posts` 상태 확인**(이미 published면 skip) — 재시도가 "컨테이너는 만들었는데 publish 직전에 죽은" 경우까지 커버.
- **데드레터(끝내 실패)**: `attempts` 소진 시 잡은 `failed`로 남고, `worker.on('failed')`에서 사용자 알림 + 검수피드 표시 + Sentry 기록. → **조용한 실패가 없다.**

### 2.3 웹훅 처리 (댓글 → DM) — "받기"와 "처리"를 분리
웹훅의 함정: Meta는 우리 서버에 댓글 이벤트를 POST하고 **빠른 200 응답**을 기대한다. 응답이 늦으면 Meta가 재전송/구독 해제를 한다. 그래서:

```
[Meta] 댓글 발생 → 우리 /webhook(Vercel API)로 POST
   │  → 검증(서명 확인)만 하고 곧장 BullMQ 큐에 "DM 잡" 넣고 즉시 200 반환
   ▼
[DM 워커]  키워드 매칭 → Private Reply(1회/7일) 발송 → leads/dm_logs 기록
           (이미 보냈으면 멱등키로 skip — 중복 DM 금지)
```
→ **받는 즉시 큐에 던지고 200을 돌려주는 패턴**이 표준. 무거운 일(매칭·DM·DB)은 워커가 뒤에서. 이게 BullMQ를 웹훅에 쓰는 진짜 이유다.

### 2.4 cron 주기 잡 (인사이트 수집)
BullMQ에 내장된 **반복 잡(repeatable/JobScheduler)** 기능을 쓰거나, 호스팅의 cron이 작은 엔드포인트를 때려 잡을 투입한다.
```ts
// 매일 새벽 3시: 활성 채널마다 "수집 잡"을 큐에 뿌림
await insightsQueue.add('collect', {}, {
  repeat: { pattern: '0 3 * * *' }   // cron 문법: 매일 03:00
});
// 워커가 채널 목록을 돌며 Insights 호출 → metrics_daily 저장
```
> "cron이 깨우면 → BullMQ가 채널별로 잘게 쪼개 처리"라 한 채널이 실패해도 나머지에 영향 없음.

## 3. 구현 가능성 평가 (현실성) — 결론: 충분히 된다

PO가 가장 걱정하는 부분. 솔직하게 난이도와 리스크를 분해한다.

### 3.1 왜 "어렵지 않다"고 보는가 (근거 5)
1. **표준·성숙 기술.** BullMQ는 Node 생태계에서 가장 널리 쓰는 작업큐 중 하나. 예약·재시도·동시성·반복잡이 **기본 기능으로 내장** → 우리가 신뢰성 엔진을 발명하지 않는다.
2. **핵심 난관은 이미 통과.** 제일 무서운 건 "인스타 2단계 발행이 코드로 진짜 되나"였는데 **PoC에서 성공**([기술스택_개발계획 §0](Kup_기술스택_개발계획.md)). BullMQ는 그 검증된 로직을 **감싸는 껍데기**일 뿐.
3. **언어 통일.** 워커도 TypeScript → 웹앱과 같은 코드/타입/SDK 공유. 새 언어·런타임 학습 없음.
4. **운영 부담이 작다.** "별도 시스템(n8n)을 또 운영"하는 게 아니라, **어차피 만드는 백엔드에 큐 라이브러리 하나 추가** + Redis 1개. 관리 표면이 n8n보다 *좁다*.
5. **점진적 도입 가능.** Phase 5에서 발행 워커 하나만 먼저([개발계획 §9](Kup_기술스택_개발계획.md)). 웹훅/수집은 그 뒤에 같은 패턴 복붙.

### 3.2 난이도 솔직 분해
| 작업 | 난이도 | 비고 |
|---|---|---|
| 큐 만들고 잡 add/워커 실행 | 하 | 라이브러리 기본. 튜토리얼 수준 |
| PoC 발행 로직을 워커로 이식 | 중하 | 로직은 있음. "잡 형태"로 재배치 |
| 멱등·중복발행 방지 | **중** | ★ 가장 신경 쓸 곳. 설계로 해결(§2.2) |
| 재시도/백오프/실패알림 | 하 | 옵션 + 이벤트 핸들러 |
| 워커 호스팅·배포·상시구동 | 중하 | Railway/Render에 올리면 됨(§4) |
| 웹훅 빠른 응답+비동기 처리 | 중 | "받고 큐에 던지기" 패턴 |
| Redis 연결 안정성·재연결 | 중 | 설정값 몇 개(§6 체크리스트) |

→ **전체적으로 "중하~중".** 미지의 R&D가 아니라 **정형 패턴의 조립**. 바이브코딩으로도 충분히 도달 가능하되, **멱등 처리와 Redis 설정값**만 정확히 짚으면 됨(§6).

### 3.3 실패할 수 있는 지점 (솔직한 리스크)
- **중복 발행** — 멱등 안 짜면 재시도가 같은 글을 두 번 올린다. → §2.2로 차단(필수).
- **Redis 설정 미스** — BullMQ는 `maxRetriesPerRequest: null` 등 특정 연결 설정을 요구. 안 맞추면 워커가 죽거나 잡이 멈춤. → §6 체크리스트.
- **워커가 꺼져 있음** — 워커는 *상시* 떠 있어야 한다. Vercel(서버리스)엔 못 올린다 → Railway/Render 같은 상시 서버 필수(§4). 배포 시 워커 재시작 중 잡은 Redis에 남아 안전.
- **비용 누수** — §0-3 함정. 잘못된 Redis 플랜 선택 → §4.

→ 모두 **알려진 함정 + 알려진 해법**. 미지의 위험이 아니다.

## 4. 비용 상세 — "어디서, 얼마가" 나가나

오케스트레이션 비용은 **딱 두 덩어리**다. (LLM 토큰비는 별개 — [개발계획 §3](Kup_기술스택_개발계획.md))

### 4.1 덩어리 ① Redis (큐 보관소)
**★ 가장 중요한 함정부터:** BullMQ는 **큐가 비어 있어도 Redis를 규칙적으로 들여다본다**(대기 잡 폴링 + 지연잡 확인 + 멈춘잡 점검). 즉 **일이 없어도 명령(command)이 계속 발생**한다. 그래서 **"명령 1건당 과금" 방식과 상극**이다.

- **Upstash Redis 가격(2026-06-24 실검증)**:
  - **무료**: 256MB / **월 50만 commands** / 월 10GB 대역폭.
  - **Pay-As-You-Go(건당)**: **10만 commands당 $0.20**.
  - **Fixed(고정요금)**: **250MB = $10/월**, 1GB = $20/월 … (명령 수 제한 없음, 초당 처리율 상한만).
- **왜 PAYG가 위험한가(수치 감)**: BullMQ 워커 1대가 *유휴 상태로도* 끊임없이 폴링하면 하루 수십만~수백만 commands가 흔하다. **무료 50만 commands는 하루 만에 소진**될 수 있고, PAYG면 *놀고 있는데도* 월 수십 달러가 샐 수 있다.
- **Upstash 공식 권고(인용)**: *"BullMQ accesses Redis regularly, even when there is no queue activity … we recommend switching to a Fixed plan to avoid increased command count and high costs in BullMQ use cases."*
- **→ 결론**: **Upstash를 쓰면 무조건 Fixed $10/월 플랜.** 무료/PAYG는 BullMQ엔 부적합(데모·실험 외).

**대안: 워커 옆에 Redis를 직접 띄우기** — Railway/Render에 Redis를 1개 추가하면 **건당 과금이 없다**(서버 임대료만, 약 $5/월~). 명령이 아무리 많아도 추가비 0. 트래픽이 커질수록 유리. 단, *내가 관리하는* 서비스가 하나 늘어남(백업·메모리). MVP 소규모면 Upstash Fixed가 더 편하고, 규모 커지면 자체 Redis가 더 쌈.

### 4.2 덩어리 ② 상시 워커 서버
워커는 **24시간 켜져** 큐를 지켜야 하므로 서버리스(Vercel) 불가 → 상시 서버 필요.

- **Render(고정요금)**: 0.5 CPU / 512MB = **$7/월** 부터(서비스당 고정). 예측 쉬움.
- **Railway(사용량 기반)**: $20/vCPU·월 + $10/GB·월. 0.5 vCPU+512MB 상시면 **약 $15/월** 추정. 저트래픽/간헐적이면 더 쌈.
- **Playwright 렌더를 이 워커에 얹으면** 메모리·CPU가 한 단계 올라가 사양 상향 필요([해설 §8.3](Kup_기술스택_해설.md)). MVP는 현행 sharp+SVG라 가벼움 → 작은 사양으로 충분.

### 4.3 월 비용 시나리오 (오케스트레이션 한정, LLM 제외)

| 단계 | Redis | 워커 서버 | 합계(월) | 비고 |
|---|---|---|---|---|
| **PoC/데모** | Upstash 무료 | (로컬/임시) | **$0** | 명령 한도 주의, 실서비스 부적합 |
| **MVP 초기(권장)** | Upstash Fixed 250MB **$10** | Render $7 또는 Railway ~$15 | **$17~25** | 가장 단순·안전한 시작 |
| **자체 Redis형** | Railway Redis ~$5 | 워커 ~$10~15 | **$15~20** | 건당 과금 0, 관리 +1 |
| **성장기** | Upstash Fixed 1GB $20 / 또는 자체 Redis | 워커 사양↑ $25+ | **$45~** | 동시성·렌더 부하↑ 반영 |

> **요지**: MVP에서 오케스트레이션 인프라비는 **월 $15~25 한 줄**로 시작 가능. **변동 위험은 Redis 플랜 선택**에 있고(PAYG 금지), 워커는 고정요금이라 예측 쉽다. **여기엔 인스타 API($0)·LLM 토큰비(별도)·DB(Supabase)·스토리지는 미포함** — 전체 인프라는 [기술설계 ⑤](Kup_기술설계.md) 참고.

## 5. 알아둬야 할 대안 (선택지를 닫지 말 것)

cron+BullMQ가 "정답"이지만, PO가 비교 우위를 설명하려면 옆 카드도 알아야 한다.

| 방식 | 한 줄 | 장점 | 단점 | 우리 입장 |
|---|---|---|---|---|
| **BullMQ + Redis** (결정안) | Redis 기반 작업큐 | 성숙·기능 풍부(예약/재시도/동시성/우선순위), Node 표준 | Redis 인프라 1개 필요, 건당과금 함정 | ✅ **채택** |
| **pg-boss** | **이미 쓰는 Postgres** 위에서 도는 큐 | **Redis 불필요**(인프라 -1, 비용 -Redis), 트랜잭션 일관성↑ | 초고처리량엔 BullMQ보다 약함, 기능 단순 | 🔶 **진지한 대안** — MVP 규모(하루 수천 잡)엔 충분. Redis 비용·운영을 통째로 없앰 |
| **호스팅 내장 cron만** | Render/Vercel Cron이 엔드포인트 호출 | 가장 단순 | 개별 재시도·지연·추적 없음 | 보조용(주기 트리거)으로만 |
| **n8n** | 비주얼 워크플로 | 빠른 시각적 검증 | 핵심 로직엔 부적합(테스트·보안·토큰) | ❌ 핵심 제외([결정문서](Kup_n8n_결정.md)) |

> **특히 pg-boss를 기억할 것**: 우리는 어차피 **Supabase Postgres**를 쓴다. pg-boss는 *그 DB 안에* 큐를 만든다 → **Redis라는 부품과 그 비용·운영·건당과금 함정을 통째로 제거**한다. MVP 규모(하루 1~2건/유저)에선 처리량이 차고 넘친다. **"Redis까지 도입할 가치가 있나?"가 의심되면 pg-boss로 시작 → 규모 커지면 BullMQ로 승급**이 합리적 경로. (BullMQ를 처음부터 쓰면 기능 여유가 있고 생태계가 크다는 게 장점.) → **재검토 포인트로 남겨둠.**

## 6. 꼭 알아야 할 함정 · 구현 체크리스트

바이브코딩으로 짤 때 **이것만 정확히 짚으면** 대부분의 사고가 막힌다.

- [ ] **Redis 플랜**: Upstash면 **Fixed**(건당 과금 PAYG/무료 금지). 또는 자체 Redis. → §4.1
- [ ] **연결 설정**: BullMQ 워커의 Redis 클라이언트는 `maxRetriesPerRequest: null` 필요(안 하면 블로킹 명령에서 에러). Upstash는 TLS 연결 문자열 사용.
- [ ] **멱등 2겹**: ① `jobId`로 등록 중복 차단 ② 워커 안에서 발행 전 상태 확인 → **중복 발행 0**. (§2.2)
- [ ] **웹훅은 받고-즉시-200**: 무거운 처리는 큐로. Meta 서명 검증 필수. (§2.3)
- [ ] **워커는 상시 서버에**(Vercel 불가). 배포·재시작 중 잡은 Redis/Postgres에 남아 유실 안 됨.
- [ ] **데드레터 처리**: `attempts` 소진 잡 → 사용자 알림 + 검수피드 + Sentry. 조용한 실패 금지.
- [ ] **동시성(concurrency)**: 인스타 발행 레이트리밋(계정당 24h ~25건)·API 시간당 한도 고려해 워커 동시 처리 수 제한.
- [ ] **모니터링**: 큐 적체/실패율 대시보드(BullMQ용 보드 or Sentry). "웹훅 누락·발행 실패"를 눈으로 봐야 함([리스크 등록부](Kup_기술설계.md) ⑩).
- [ ] **토큰 취급**: 워커가 발행 직전 복호화, 메모리에서만 사용, 로그에 토큰 금지. (SPEC 2.2)

## 7. 직접 검증하는 법 (싸게, 키만 있으면)

"진짜 되는지" 의심을 코드로 닫는 최소 PoC. 이미 발행은 코드로 성공했으니 **큐 껍데기만** 씌우면 됨.

1. **로컬 Redis 1줄로 띄우기**(Docker) → BullMQ 큐 1개 + 워커 1개 작성.
2. **딜레이 잡 테스트**: "30초 뒤 콘솔에 찍기" 잡 등록 → 30초 뒤 워커가 실행하는지. (예약의 본질 확인)
3. **재시도 테스트**: 일부러 실패하는 잡 → 백오프로 3번 재시도 후 `failed` 가는지.
4. **발행 이식**: 검증 PoC(비공개)의 2단계 발행을 워커 함수 안으로 → 큐에 넣어 실제 발행되는지.
5. **멱등 테스트**: 같은 `jobId`로 두 번 add → 한 번만 실행되는지.
6. **(비용 감 잡기)** Upstash 무료 DB로 워커를 1시간 유휴로 켜두고 **commands 카운터 증가**를 눈으로 확인 → "폴링이 명령을 먹는다"를 체감 → Fixed 플랜 필요성 실증.

> 1~5는 반나절 안에 가능. 6은 비용 함정을 *눈으로* 확인하는 단계 — PO가 직접 보면 가장 확실하다.
