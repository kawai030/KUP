# Kup — n8n 사용 여부 결정 문서

> **목적**: "오케스트레이션(예약·발행·웹훅·재시도)을 n8n으로 할 것인가, 코드로 할 것인가"를 단독으로 정리·결정하고, 그 근거를 보존한다.
> **결론(먼저)**: **핵심 경로(발행·예약·웹훅·재시도)는 코드(BullMQ+cron)로 한다. n8n은 핵심에서 빼고, "초기 1회성 검증"과 "비핵심 내부 자동화"에만 선택적으로 쓴다.**
> **기준일**: 2026-06-24 · 관련: [기술설계 §7](Kup_기술설계.md), [기술스택_개발계획 §8](Kup_기술스택_개발계획.md), [기술스택 해설 §5](Kup_기술스택_해설.md)

---

## 1. n8n이 뭔가

- **n8n**: "노드(상자)를 선으로 이어" 자동화 흐름을 만드는 **비주얼 워크플로 자동화 도구**(코딩 거의 없이). Zapier·Make의 오픈소스/셀프호스팅 버전 격.
- 예: `[웹훅 수신] → [조건 분기] → [HTTP 요청] → [메일 발송]` 을 화면에서 드래그로 연결.
- 장점: **빠르게 만들고 눈으로 흐름이 보인다.** 비개발자도 간단한 연동을 짤 수 있다.

## 2. 우리가 "오케스트레이션"으로 해야 하는 일

PRD/SPEC가 n8n으로 적었던 영역 = 다음의 **시간·상태·신뢰성**이 걸린 작업들:
- **예약 발행**: 승인된 시안을 정해진 시각에 인스타 3단계로 게시
- **재시도**: 발행 실패 시 백오프 재시도, 계속 실패 시 알림
- **웹훅 처리**: 댓글 이벤트 수신 → 키워드 매칭 → Private Reply DM(7일·1회 정책)
- **주기 수집**: 인사이트(팔로워 순증 등) 정기 수집

→ 이건 단순 글루(glue)가 아니라 **제품의 핵심 비즈니스 로직**이다.

## 3. 결정: 핵심은 코드(BullMQ), n8n 아님 — 이유 5가지

| # | 기준 | n8n | 코드(BullMQ+cron) | 우리에게 왜 중요한가 |
|---|---|---|---|---|
| 1 | **버전관리·테스트** | 워크플로가 GUI 상태라 git 추적·자동테스트 약함 | 코드라 git·CI·유닛테스트 ✅ | 바이브코딩이라 변경 이력·회귀 방지가 생명 |
| 2 | **에러추적·재시도 제어** | 제한적, 세밀한 백오프·멱등 처리 어려움 | Sentry 연동·정교한 재시도 ✅ | 발행/DM은 실패하면 사용자 신뢰 직격 |
| 3 | **민감 상태/토큰 취급** | 인스타 토큰·검수상태가 n8n을 거침 → **보안 표면 확대** | 우리 백엔드 안에서만 처리 ✅ | 토큰 유출 = 최악의 사고(암호화 원칙과 충돌) |
| 4 | **운영 부담** | **별도 시스템 1개**를 더 설치·보안·확장·업데이트 | 어차피 만드는 백엔드에 큐만 추가 | 3인 팀, 운영할 시스템 최소화 |
| 5 | **복잡 로직 표현** | 분기·루프·상태머신이 많아지면 비주얼이 오히려 난잡 | 코드가 명확·유지보수 쉬움 | 발행=3단계 폴링+재시도+멱등, 단순하지 않음 |

**한 줄 근거**: 핵심 로직을 비주얼 도구에 넣으면 *테스트·보안·추적*을 잃는다. 우리는 어차피 백엔드를 만드니, 거기에 **작업큐(BullMQ) 하나 추가**가 더 안전하고 깔끔하다.

## 4. 그럼에도 n8n이 쓸모 있는 경우 (선택적 사용 허용)

1. **초기 스파이크(1회성 검증)**: 코드 짜기 전, 인스타 발행 흐름을 n8n으로 빠르게 한 번 성공시켜 "된다"는 자신감·이해를 얻고 → 그 로직을 코드로 옮긴다.
   - 단, **이미 검증 PoC(비공개)에서 발행을 코드로 성공**시켰으므로 이 단계는 사실상 통과 — n8n 없이도 검증됨.
2. **비핵심 내부 운영 자동화**: 슬랙 알림, 주간 리포트 메일, 마케팅 글루 등 **토큰·검수와 무관한** 잡일. 여기엔 n8n을 써도 무방(리스크 낮음).

## 5. 경계선 (헷갈리지 않게)

| 작업 | 어디서 | 이유 |
|---|---|---|
| 예약 발행 / 재시도 | **코드(BullMQ)** | 핵심·민감·신뢰성 |
| 댓글 웹훅 → DM | **코드** | 토큰·정책(7일/1회)·멱등 |
| 인사이트 주기 수집 | **코드(cron→BullMQ)** | 데이터 정합성 |
| 사내 슬랙 알림 | n8n 가능 | 비핵심 글루 |
| 출시 전 발행흐름 1회 시연 | n8n 가능(선택) | 스파이크 — 이미 코드로 통과함 |

## 6. 결정 로그 (Decision Log)

- **결정**: 프로덕션 backbone = 코드(BullMQ + cron). n8n은 핵심 경로 제외, 비핵심·스파이크 한정 선택 사용.
- **상태**: 확정 (기술설계 §7 / 개발계획 §8과 일치).
- **재검토 트리거**: 비개발 인력만으로 운영해야 하는 상황이 오거나, 비핵심 자동화가 급증하면 n8n 도입 범위 재검토.
- **영향**: PRD/SPEC의 "n8n 오케스트레이션" 표현을 "코드 기반 스케줄러/큐"로 대체.
