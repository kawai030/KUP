# Kup — 기술 설계 문서 (Technical Design)

> **목적**: PRD/SPEC를 "실제로 어떻게 만드는가"로 번역한 문서. 개발 비전공자도 전체 그림과 각 기능의 작동 원리, 필요한 기술·API를 이해할 수 있도록 작성.
> **작성 기준**: 2026년 6월 인스타그램/Meta API 정책 실검증 반영.
> **읽는 순서**: ① 결론 요약 → ② 전체 아키텍처 → ③ 인스타 API 현실 → ④ 기술 스택 → ⑤ 외부 서비스/비용 → ⑥ 기능별 상세 로직 → ⑦ n8n 판단 → ⑧ 데이터 모델 → ⑨ 실현가능성 테스트 계획 → ⑩ 리스크.

---

## ① 결론 요약 (1분 브리핑)

1. **핵심 기능 전부 공식 API로 구현 가능하다.** 카드뉴스(캐러셀)·릴스 자동 발행, 댓글→DM 리드마그넷, 성과 수집 모두 Meta 공식 인스타그램 API가 지원한다. 비공식·해킹성 자동화가 필요 없다 → 계정 정지 리스크 낮음. PRD의 "공식 API만" 원칙이 기술적으로 성립한다.

2. **단, "앱 리뷰"가 최대 시간 리스크다.** 실서비스로 남의 인스타 계정에 발행/DM 하려면 Meta 앱 심사(2~4주, 권한별 화면녹화 제출, 사업자 인증)를 통과해야 한다. → **앱 리뷰는 지금 당장 착수**해야 출시 일정이 안 밀린다. 반대로, **개발/테스트는 앱 리뷰 없이 지금 바로 가능**하다(내 계정/테스터 계정 한정, '개발 모드').

3. **n8n은 핵심 경로에 넣지 말 것을 권장.** 발행·예약·재시도·웹훅은 제품의 핵심 비즈니스 로직이라 코드로 두는 게 안정적·관측가능·테스트가능하다. n8n은 ① 초기 인스타 발행 흐름을 빠르게 검증하는 "스파이크 도구" ② 내부 운영 자동화 정도로만 선택적으로 쓴다. (자세한 근거는 ⑦장)

4. **MVP 기술 난이도 순서**: (쉬움) Claude 카피 생성 → 카드뉴스 이미지 렌더 → (중간) 인스타 OAuth/발행 → (중간) 댓글 웹훅→DM → (어려움/뒤로) 릴스 영상 생성. **카드뉴스 우선 전략이 기술적으로도 옳다** — 영상 생성이 제일 어렵고 비싸기 때문.

## ② 전체 아키텍처 (한눈에)

PRD의 "3-레이어 분리"를 실제 시스템으로 펼치면 이렇게 된다.

```
[사용자 — 웹앱(브라우저)]
        │  로그인, 컨셉 잠금, 시안 검수/승인, 대시보드
        ▼
┌─────────────────────────────────────────────────────────┐
│  우리 백엔드 (제품의 두뇌)                                  │
│                                                          │
│  · API 서버      ── 화면 요청 처리, 인증                    │
│  · 생성 오케스트레이터 ── 컨셉→전략→카피→카드 렌더 묶기       │
│  · 검수/리스크 엔진 ── 저위험/고위험 분류, 플래그            │
│  · 스케줄러 + 작업큐 ── "예약 시각에 발행" 같은 시간 작업      │
│  · 토큰 보관소(암호화) ── 인스타 액세스 토큰 안전 보관         │
└───┬───────────────┬────────────────┬─────────────────────┘
    │               │                │
    ▼               ▼                ▼
[생성 레이어]    [영상 레이어]      [발행·데이터 레이어]
Claude API      Creatomate 등     Meta 인스타그램 API
(기획·카피·       (릴스 렌더,        (캐러셀/릴스 발행,
 스크립트)        보조·후순위)        성과수집, 댓글웹훅, DM)
                                       ▲
                                       │ 웹훅(댓글 발생 알림)
                                  [인스타그램 플랫폼]
```

핵심: **Claude는 글자만 만든다. 인스타에 실제로 올리고 데이터를 가져오는 건 오직 Meta API다.** 우리 백엔드가 이 둘을 묶고, 시간(예약)과 안전(검수)을 책임진다.

## ③ 인스타그램 API의 "현실" (가장 중요한 장)

서비스 전체가 여기에 달려 있으므로 사실관계를 정확히 정리한다. (출처는 문서 맨 아래)

### 3.1 어떤 API를 쓰나 — "Instagram API with Instagram Login"
- 2024년 이후 Meta가 밀고 있는 방식. **페이스북 페이지 연결이 필요 없다.** 인스타그램으로 바로 OAuth 로그인 → 그 계정의 액세스 토큰을 받는다.
- 우리 케이스(개인/소상공인이 자기 인스타를 우리 서비스에 연결)에 가장 적합하다. 예전 방식(Facebook 페이지 경유 Graph API)보다 사용자 온보딩이 훨씬 간단하다.
- **비용: $0** (Meta 공식 API 호출 자체는 무료. 인프라/AI 비용만 우리가 부담).
- 전제: 사용자의 인스타가 **프로페셔널 계정(비즈니스/크리에이터)** 이어야 한다. 개인 계정은 API 불가 (전환은 30초·무료).

### 3.2 콘텐츠 발행 — 카드뉴스/릴스 둘 다 가능
- **3단계 컨테이너 발행 모델**: ① 미디어 컨테이너 생성 → ② 처리 완료(FINISHED) 확인 → ③ 게시. (SPEC 2.3과 일치)
- **카드뉴스 = 캐러셀**: 이미지 여러 장(최대 10장)을 한 게시물로. **캐러셀 1개 = 발행 1건으로 카운트.** → 카드뉴스에 유리.
- **릴스**: 영상 URL을 컨테이너로 올려 발행 가능.
- 필요 권한: `instagram_business_content_publish`.
- **발행 한도(레이트리밋)**: 계정당 **24시간에 약 25건** 발행(피드 기준). 우리 서비스(하루 1~2건 발행)엔 차고 넘침. API 일반 호출은 시간당 약 200건.

### 3.3 댓글 → DM 리드마그넷 — 공식 기능으로 가능
- 메커니즘: 우리 게시물에 사용자가 댓글 → Meta가 **웹훅(comment 이벤트)** 으로 우리 서버에 알림 → 우리가 그 사람에게 **'프라이빗 리플라이(1회 DM)'** 발송.
- **정책 한도(2026)**: 댓글 단 사람에게 **딱 1번**, **7일 이내** DM 가능. (리드마그넷 용도에 정확히 맞는 합법 경로 = "Private Reply")
- 그 사람이 답장하면 24시간 대화창이 열려 추가 대화 가능.
- 필요 권한: `instagram_manage_comments`(댓글 읽기/관리) + `instagram_manage_messages`(DM 발송).
- **금지선**: 먼저 말 안 건 사람에게 콜드 DM, 자동 팔로우는 불가 → PRD의 "MUST NOT"과 정확히 일치. 정책이 우리 설계를 강제로 안전하게 만들어준다.

### 3.4 성과 데이터 수집
- `instagram_business_basic` 등으로 팔로워 수, 도달(reach), 노출, 댓글, 인사이트 수집 가능 → 대시보드/ KPI 계측 가능. (단 지표별로 집계 주기·지연 있음)
- 주의: `follows_and_unfollows`(순증) 같은 일부 지표는 계정 규모/지표 종류에 따라 제공 범위가 다름 → 테스트 단계에서 실제 어떤 필드가 내려오는지 확인 필요(테스트 항목에 포함).

### 3.5 앱 리뷰 — "출시 게이트", 지금 착수해야 함
- 위 권한 4종(`instagram_business_basic`, `instagram_manage_comments`, `instagram_manage_messages`, `instagram_business_content_publish`)을 **남의 계정에 쓰려면 Meta 앱 심사 통과 필수.**
- 절차: 사업자/비즈니스 인증 → 권한마다 **사용 장면 화면녹화(스크린캐스트)** 제출 → 테스트 계정 제공 → 심사. **소요 2~4주, 반려 흔함**(녹화 불충분이 주원인).
- **하지만 개발 모드에선 앱 심사 없이** 앱에 역할(관리자/개발자/테스터)이 부여된 계정 한정으로 위 기능 전부 테스트 가능. → **우리 팀 계정으로 지금 바로 PoC 가능.**
- 결론: **개발/검증은 즉시 시작, 앱 리뷰는 병렬로 미리 준비**(데모 영상 찍을 수 있을 만큼 기능이 돌아가면 바로 제출).

## ④ 기술 스택 (추천 + 이유)

> 원칙: 3인 팀 + 비개발자 PO 환경 → **한 가지 언어(TypeScript)로 통일**해 인력 효율을 높이고, **관리형 서비스(managed)** 를 적극 써서 운영 부담을 줄인다.

| 영역 | 추천 | 왜 |
|---|---|---|
| 웹 프론트엔드 | **Next.js (React) + TypeScript + Tailwind** | 검수 UI·대시보드를 빠르게. 한국 개발자 풀 넓음 |
| 백엔드 API | **Next.js API / Node.js (TypeScript)** | 프론트와 같은 언어. Meta·Claude SDK 모두 JS 지원 좋음 |
| 데이터베이스 | **PostgreSQL (Supabase 관리형)** | 안정적 관계형 DB. Supabase가 인증·스토리지·DB를 한 번에 → 3인 팀 속도↑ |
| 로그인(우리 서비스) | **Supabase Auth / Clerk** | 우리 서비스 자체 회원 로그인 (인스타 OAuth와 별개) |
| 백그라운드 작업·예약 | **BullMQ (Redis 기반 작업큐) + cron** | "예약 발행, 재시도, 웹훅 처리" 코어 로직. n8n 대체 (⑦장) |
| 파일 저장 | **S3 / Cloudflare R2 / Supabase Storage** | 렌더된 카드 이미지·영상 보관 |
| 토큰 암호화 | **libsodium / KMS + DB 암호화 컬럼** | 인스타 토큰은 평문 저장 금지(SPEC 2.2) |
| AI 생성 | **Claude API (Anthropic SDK)** | 기획·카피·스크립트. 최신 모델(claude-opus-4-8 / sonnet) |
| 카드뉴스 렌더 | **HTML/CSS 템플릿 → 헤드리스 렌더(Playwright) 또는 Satori** | 브랜드 컬러·템플릿을 코드로 정밀 제어, 비용 저렴 (4.1 참고) |
| 영상(릴스) | **Creatomate(템플릿 렌더) / ZapCap(자막)** | 보조 기능. 후순위. 외부 API로 위임 |
| 호스팅 | **Vercel(프론트) + Railway/Render/Fly(워커)** | 워커는 Playwright·작업큐 돌릴 상시 서버 필요 |
| 모니터링 | **Sentry + 로그(Logtail 등)** | 발행 실패·웹훅 누락 추적(운영 필수) |

### 4.1 카드뉴스(캐러셀) 이미지를 "어떻게" 만드나 — 중요 기술 결정
인스타 발행 API는 **완성된 이미지 URL**을 받는다. 즉 "디자인된 카드 이미지"를 우리가 먼저 만들어 어딘가(스토리지)에 올려두고, 그 URL을 API에 넘겨야 한다. 만드는 방법 3가지:

1. **(추천) HTML/CSS 템플릿 + 헤드리스 렌더**: 카드 디자인을 웹페이지처럼 HTML/CSS로 만들고, Playwright(보이지 않는 브라우저)로 PNG 스크린샷. 브랜드 컬러·폰트·레이아웃을 픽셀 단위로 통제, 비용 거의 0. 변형안(2~3개) 생성도 쉬움.
2. **관리형 이미지 API**: Bannerbear / Placid / Creatomate. 템플릿을 그쪽 대시보드에서 만들고 데이터만 API로 주입. 빠르지만 월 구독비·디자인 자유도 제약.
3. **AI 이미지 생성(텍스트→이미지)**: MVP에선 **비권장**. 손가락·글자 왜곡 등 아티팩트 → 고위험 검수 폭증, PRD의 "신뢰" 포지션과 충돌. 사진이 꼭 필요하면 사용자 업로드 자산(B 모드) 또는 스톡으로.

→ **MVP 권장**: 타이포그래피 중심 카드뉴스(텍스트+브랜드 컬러+심플 그래픽)를 HTML/CSS 템플릿으로 렌더. AI는 "글(카피)"만, 디자인은 "템플릿"이 책임진다. 이러면 품질이 안정적이고 검수가 진짜 5초가 된다.

## ⑥ 기능별 상세 로직 플로우

각 기능을 "입력 → 우리 시스템이 하는 일(단계별) → 어떤 API 호출 → 무엇을 저장 → 출력"으로 적는다.

### 6.1 온보딩 & 컨셉 잠금
- **입력**: 사용자가 주제·타깃 입력, 모드 선택(생성형 A / 자산기반 B / 증강 C).
- **단계**:
  1. 우리 서비스 회원가입/로그인(Supabase Auth).
  2. **인스타 연결(OAuth)**: "인스타로 로그인" 버튼 → Meta 동의화면 → 코드 수신 → 우리 백엔드가 단기토큰 받기 → **장기토큰(60일)으로 교환** → **암호화해서 DB 저장**. 비번은 절대 안 받음(SPEC 2.2).
     - 비즈니스 계정 아니면 → "프로페셔널 전환 필요" 안내, 발행 잠금 유지(SPEC 3.1 실패 시나리오).
  3. **컨셉 제안**: Claude에 주제·타깃·모드를 주고 → 페르소나·톤·비주얼 방향·콘텐츠 기둥(주제 카테고리들)·게시 주기를 **여러 안으로 제안**.
  4. 사용자가 1개 **확정(잠금)** → `channel_config`로 저장. 이후 모든 시안 생성의 제약(시스템 프롬프트)으로 자동 상속.
- **API**: Meta OAuth, Claude API.
- **저장**: `channels`(연결 계정), `channel_config`(잠긴 컨셉), `ig_tokens`(암호화 토큰).

### 6.2 콘텐츠 생성 엔진 (3모드, 한 엔진)
- **트리거**: 스케줄(매일/주기) 또는 사용자가 "시안 만들기".
- **단계**:
  1. (선택) **주제 추천·탐색**: "핫/팔로워 잘 모이는/조회수 높은" 주제 제안. → MVP 구현 방식: 컨셉 기반으로 Claude가 후보 주제 생성 + (가능하면) 자사 누적 성과 데이터로 가중. *주의: 인스타 API는 "남의 잘 되는 게시물" 데이터를 주지 않는다 → 외부 트렌드 소스나 우리 내부 성과로 근사.*
  2. **전략 생성**: 선택 주제 → Claude가 후킹 앵글·구성(카드 장수, 슬라이드별 메시지) 설계.
  3. **카피 생성**: 슬라이드별 텍스트 + 캡션 + 해시태그를 Claude가 생성. (잠긴 컨셉을 제약으로)
  4. **카드 렌더**: 슬라이드 텍스트를 HTML/CSS 템플릿(브랜드 컬러)에 주입 → Playwright로 PNG 변환 → 스토리지 업로드 → URL 확보. (변형안 2~3개 옵션)
  5. **모드별 차이**:
     - A(생성형): 1~4 그대로.
     - B(자산기반): 사용자가 올린 제품/매장 이미지를 카드 템플릿에 합성.
     - C(증강): 보유 영상 → (MVP는 자막/컷 정도, 완전생성은 M2) → 릴스 스크립트는 Claude.
  6. **AI 자가 점검**: Claude가 생성물에서 미검증 주장·규제 표현·이미지 의심점을 **스스로 플래그** → 검수 단계로 넘김.
- **API**: Claude API, (렌더는 내부 Playwright), (B모드 이미지 처리).
- **저장**: `drafts`(시안: 텍스트·이미지URL·플래그·리스크등급).

### 6.3 휴먼 인 더 루프 검수 (차별화 핵심)
- **입력**: 생성된 시안 + 자가점검 플래그 + 리스크 등급.
- **리스크 분류(검수 엔진)**: 규칙 + Claude 판단으로 **저위험/고위험** 라벨.
  - 고위험 트리거: 미검증 사실/수치, 실존 인물, 브랜드·효능 주장, 규제 업종(금융=권유표현, 의료 등).
- **검수 UI(5초 결정)**:
  - 플래그된 부분만 강조 표시(빈 검수 금지).
  - **자연어 수정**: "더 캐주얼하게" 입력 → 캡션 직접편집 X → Claude 재생성.
  - **변형안 선택**: 2~3개 중 택1.
  - 승인 / 거부 / 수정요청.
- **라우팅**:
  - 고위험: **사람 검수 없이 발행 불가(MUST NOT)**, 체크박스 명시 확인 필수.
  - 저위험: 원탭 승인, (신뢰 누적 후) 조건부 자동 흐름.
- **오토파일럿**: 동일 저위험 유형 N회 연속 승인 시 자동 흐름 제안. **킬 스위치 + 사후 검수 피드(게시 후 소급 삭제)** 상시 유지.
- **저장**: `reviews`(누가·언제·무엇을 승인/수정 = 감사 추적), `drafts.status` 갱신.

### 6.4 발행 (스케줄러 + Meta API 2단계)
- **트리거**: 승인 + 예약 시각 도달.
- **단계(작업큐 잡)**:
  1. 스케줄러가 예약 시각에 "발행 잡"을 작업큐에 투입.
  2. 잡 실행: **AI 라벨 부착** + 캡션 최종화.
  3. **컨테이너 생성**: 캐러셀이면 슬라이드 이미지 URL들로 자식 컨테이너 → 부모 컨테이너. (릴스면 영상 URL)
  4. **상태 폴링**: 컨테이너가 `FINISHED` 될 때까지 확인(영상은 인코딩 시간 필요).
  5. **게시(publish)** 호출 → 게시물 ID 수신.
  6. 실패 시 **재시도(백오프)**, 계속 실패면 사용자 알림 + 검수 피드에 표시.
- **API**: Meta 컨테이너 생성/상태/게시 엔드포인트.
- **저장**: `posts`(게시물ID·시각·상태), 이벤트 로그(활성화: "첫 자동 발행" = Aha).

### 6.5 코멘트 → DM 리드마그넷
- **사전 설정**: 사용자가 게시물에 "키워드→DM 내용(링크/자료)" 규칙 설정.
- **단계**:
  1. Meta **웹훅 구독**(comments). 누군가 댓글 → Meta가 우리 서버로 이벤트 POST.
  2. 우리 서버가 키워드 매칭 확인.
  3. 매칭 시 **Private Reply(1회 DM)** API 호출 → 그 사용자에게 자료/링크 DM. (7일·1회 정책 준수)
  4. (선택) 공개 댓글로 "DM 확인하세요" 답글.
  5. DM 로그 기록(리드 계측).
- **API**: Meta 웹훅, 댓글 API, 메시지(Private Reply) API.
- **인프라 주의**: 웹훅은 **공개 HTTPS 주소**가 필요(개발 땐 ngrok/cloudflared로 로컬 노출).
- **저장**: `leads`(누구·언제·어떤 게시물·전환), `dm_logs`.

### 6.6 분석 & KPI 대시보드
- **단계**:
  1. 주기적 잡이 Meta Insights에서 도달·노출·팔로워수·댓글 등 수집.
  2. 우리 내부 이벤트(가입·연동·승인·발행·리드)와 합쳐 KPI 산출.
  3. North Star(주간 팔로워 순증), 퍼널, 가드레일(무수정 승인율·검수시간·제재율) 시각화.
- **API**: Meta Insights. **저장**: `metrics_daily`, `events`.

## ⑦ n8n — 필요한가? (명확한 판단)

**결론: 핵심 경로에는 쓰지 말고, 코드 기반 작업큐(BullMQ)로 만들 것을 권장한다. n8n은 선택적 보조 도구.**

### 근거
- PRD/SPEC는 n8n을 "오케스트레이션(예약·발행·웹훅·재시도)"으로 적었지만, 이건 **제품의 핵심 비즈니스 로직**이다. 핵심 로직은:
  - 코드로 둬야 **버전관리·자동테스트·에러추적(Sentry)·세밀한 재시도 제어**가 된다.
  - 검수 상태·리스크 등급·토큰 같은 **민감/복잡한 상태**를 다룬다 → 비주얼 워크플로보다 코드가 안전.
- n8n을 핵심에 넣으면 **별도 시스템 1개를 더 운영·보안·확장**해야 하고, 인스타 토큰이 n8n을 거치며 **보안 표면이 넓어진다.**
- 우리는 어차피 백엔드를 만든다 → 거기에 작업큐(BullMQ+Redis) 하나 추가가 더 깔끔하다.

### 그래도 n8n이 쓸모 있는 경우 (선택)
1. **초기 스파이크**: 인스타 발행 흐름을 코드 짜기 전에 n8n으로 빠르게 한 번 성공시켜 보고(자신감/검증), 그 로직을 코드로 옮긴다.
2. **내부 운영 자동화**: 슬랙 알림, 리포트 메일, 비핵심 글루 작업.

> 즉, **"n8n으로 빠르게 테스트는 OK, 프로덕션 backbone은 코드"**. PRD의 n8n 의존을 "코드 기반 스케줄러/큐"로 대체하는 설계 결정을 제안한다. (Decision Log에 추가 권장)

## ⑨ 실현가능성(PoC) 테스트 계획 — "간단하게 먼저"

목표: 큰돈·앱리뷰 전에 **각 기술 조각이 진짜 되는지** 싸게 확인. 위험·가치 순으로 정렬.

### 테스트 0 — 환경 준비 (1회)
- Node.js 설치 여부 확인, Claude API 키 발급, Meta 개발자 계정·앱(타입: Business) 생성, **테스트용 인스타 프로페셔널 계정**.

### 테스트 1 — Claude 카피 생성 (제일 쉬움, 지금 가능)
- 컨셉(주제·타깃)을 주고 카드뉴스 슬라이드 텍스트+캡션+해시태그가 잘 나오는지.
- **검증 포인트**: 톤 제약이 먹히나, 자가점검 플래그가 동작하나. → **로컬에서 즉시 가능.**

### 테스트 2 — 카드뉴스 렌더 (지금 가능)
- 슬라이드 텍스트 → HTML/CSS 템플릿 → Playwright로 PNG 변환 → 브랜드 컬러 적용 확인.
- **검증 포인트**: 결과물이 "그대로 올려도 될" 품질인가. → **로컬에서 즉시 가능.**

### 테스트 3 — 인스타 OAuth + 발행 (앱리뷰 불필요, 개발 모드)
- 내 테스트 계정 연결 → 단기→장기 토큰 교환 → 카드 이미지 캐러셀 **실제 발행**까지.
- **검증 포인트**: 3단계 컨테이너 발행이 실제로 게시되나, 토큰 흐름 OK인가.
- 필요: Meta 앱 설정 + 테스트 계정 토큰. (내가 스크립트 미리 작성 가능, 토큰만 넣으면 실행)

### 테스트 4 — 댓글 웹훅 → DM (개발 모드)
- 공개 HTTPS(ngrok) + 웹훅 구독 → 테스트 계정으로 댓글 → 이벤트 수신 → Private Reply DM 발송.
- **검증 포인트**: 웹훅이 진짜 오나, 1회 DM이 실제 가나, 7일 정책 동작.

### 테스트 5 — 성과 데이터 필드 확인 (개발 모드)
- Insights에서 어떤 지표(팔로워 순증 등)가 실제로 내려오는지 필드 점검. (KPI 산출 가능성 확인)

### 테스트 6 (후순위) — 릴스 영상 렌더
- Creatomate 등으로 템플릿 영상 1개 만들어 릴스 발행까지. MVP 막판/M2.

> **권장 진행 순서**: 1·2를 먼저(개발자 없이도 내가 작은 스크립트로 시연 가능) → 3·4가 진짜 관문 → 통과하면 앱 리뷰 제출 자료(데모 영상)가 자동으로 생긴다.
