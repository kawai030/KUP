## 0. 현재 위치 (PoC 결과 반영)

이미 끝난 것:
- ✅ Meta 개발자 앱 + 인스타/페북 계정 연동, **자동 발행(캐러셀) 실제 테스트 성공**
- ✅ **댓글→DM 자동화** 테스트 성공
- ✅ 카드뉴스 렌더(JSON→PNG) 동작 (검증 PoC → `lib/render/deck-renderer.ts`로 이식)
- ⚠️ 단, 카드뉴스 콘텐츠는 **수기/Claude Code로 직접 작성**했고, **LLM API를 코드로 연동한 적은 없음**

따라서 지금 필요한 두 갈래:
1. **LLM API 검증 & 선택** — Claude/OpenAI/Gemini를 같은 작업으로 돌려 **시간·비용·품질** 비교 (5장)
2. **기술스택 확정 & 개발 계획** — DB/백엔드/오케스트레이션 등 결정 (6~9장)

## 1. 결론 요약 (1분 브리핑)

1. **LLM은 "단일 고정"이 아니라 "작업별 라우팅"으로 간다.** 카드뉴스 카피 생성은 깊은 추론이 필요 없어서 *중·저가 비추론 티어*가 적합하다. 리스크 자가점검·전략 설계처럼 판단이 필요한 곳만 상위 모델. → 비용을 10배까지 줄일 수 있는 핵심 레버.
2. **벤치마크를 먼저 돌려 숫자로 정한다.** 세 공급자를 같은 프롬프트로 N회 돌려 **지연시간(p50/p95)·생성당 비용·품질 점수**를 측정한다. 추측 금지. (벤치 하니스 사양은 5.4)
3. **오케스트레이션은 n8n이 아니라 코드(BullMQ + cron)로 간다.** 발행·예약·웹훅·재시도는 제품의 핵심 비즈니스 로직이고 민감한 토큰·검수 상태를 다루므로 코드가 안전·관측가능. (기술설계 ⑦장과 동일 결론, 8장에서 확정)
4. **스택은 TypeScript 단일 언어 + 관리형 서비스.** Next.js(프론트+API) / Supabase(Postgres·인증·스토리지) / BullMQ+Upstash Redis(작업큐) / Anthropic·OpenAI·Google SDK(생성) / Playwright 또는 현행 sharp+SVG(렌더). (7장)
5. **개발 순서**: ① LLM 벤치 → ② 생성 파이프라인(컨셉→전략→카피→렌더) 코드화 → ③ 데이터모델+스토리지+OAuth 영속화 → ④ 검수 UI → ⑤ 스케줄러/발행 워커 → ⑥ 웹훅/DM/분석. (9장)

## 2. LLM 비교 — 왜 제대로 측정해야 하나

생성 비용은 **사용자 수 × 발행 빈도에 비례하는 변동비**다. 모델을 잘못 고르면 마진이 직접 깎인다. 동시에 품질(톤 준수·자가점검 정확도)이 낮으면 검수 시간이 늘어 차별화(5초 검수)가 무너진다. → **비용·속도·품질 3축을 한 번에** 봐야 한다.

핵심 인사이트:
- 카드뉴스 카피는 "긴 추론"이 아니라 "제약(잠긴 컨셉) 안에서 짧은 텍스트 N조각"을 만드는 작업. **추론(thinking) 토큰을 많이 쓰면 비용·지연만 늘고 품질 이득은 작다.** → 비추론/저effort 티어부터 시험.
- 잠긴 컨셉(페르소나·톤·기둥)은 채널마다 고정 → **프롬프트 캐싱**으로 시스템 프롬프트 입력비를 ~90% 절감 가능 (Claude/OpenAI/Gemini 모두 지원). 비용 추정에 반드시 반영.

## 4. 품질 평가 기준 (주관 점수의 객관화)

품질은 숫자가 없으면 "느낌"으로 흐른다. 생성 결과를 아래 루브릭으로 0~3점씩 채점(사람 또는 LLM-as-judge):

| 항목 | 0 | 3 |
|---|---|---|
| **톤·컨셉 준수** | 잠긴 톤 무시 | 페르소나·톤·기둥 완벽 반영 |
| **구조 유효성** | JSON 깨짐/슬롯 누락 | 스키마(cover/body/outro+caption+hashtags) 완전 충족 |
| **카피 품질** | 진부·어색 | 후킹 강하고 자연스러움 |
| **자가점검 정확도** | 위험(미검증 사실 등) 놓침 | 고위험 지점 정확히 플래그 |
| **그대로 발행 가능성** | 손봐야 함 | 수정 없이 승인 가능 |

> "무수정 승인율 60~85%"(SPEC 가드레일)와 직결. 자가점검 정확도가 낮으면 5초 검수가 깨진다 → **단순히 싼 모델이 아니라 "검수 시간을 줄이는" 모델이 이김.**

## 5. LLM 벤치마크 실행 계획

### 5.1 측정 지표
- **지연시간**: 첫 토큰까지(TTFT)·전체 완료까지, p50/p95 (N≥10회/모델/시나리오)
- **비용**: 응답의 실제 usage(입력/출력/캐시 토큰) × 단가 = 생성당 실비
- **품질**: 4장 루브릭 점수(고정 샘플 시나리오 3종에 대해)

### 5.2 고정 테스트 시나리오 (3종, A모드 기준)
1. 재테크(사회초년생) — `poc/decks/playlist-001.json` 류
2. 디저트 맛집 후기 — **고위험**(실존 매장) → 자가점검 검증용
3. 자유 주제 추천 — "뭘 올릴지 모름" 시나리오

### 5.3 공정성 원칙
- **동일 프롬프트·동일 스키마(JSON 구조화 출력)·동일 max_tokens**로 세 공급자 호출
- 각 공급자 **공식 SDK** 사용(실제 운영 지연을 반영): `@anthropic-ai/sdk`, `openai`, `@google/genai`
- 캐싱 on/off 둘 다 측정(채널 재사용 효과 확인)
- 추론(thinking/effort) on/off 둘 다 측정(카피 작업에 필요한지 판단)

### 5.4 산출물
`poc/llm-bench/` 하니스 — 입력: 시나리오 JSON, 출력: ① 모델별 결과 덱(렌더해서 눈으로 비교) ② `results.csv`(모델·시나리오·TTFT·총시간·입력/출력토큰·실비) ③ 품질 채점 시트.
→ **이 표 하나로 "카피=모델X, 판단=모델Y" 라우팅을 확정**한다.

> 이 하니스는 다음 단계로 바로 만들 수 있다(키만 넣으면 실행). 9장 Phase 1 참고.

## 7. 기술스택 확정

| 영역 | 선택 | 이유 |
|---|---|---|
| 언어 | **TypeScript** 단일 | 3인+비개발 PO, 인력 효율. 프론트/백/SDK 전부 JS 친화 |
| 프론트+API | **Next.js (App Router) + Tailwind** | 검수 UI·대시보드 빠르게. API Route로 백엔드 겸함 |
| DB | **Supabase (Postgres)** | 관계형 안정성 + 인증·스토리지 일체형 → 속도↑ |
| 자체 인증 | **Supabase Auth** | 인스타 OAuth와 별개의 서비스 로그인 |
| 작업큐/예약 | **BullMQ + Upstash Redis** | 발행·재시도·웹훅 처리. n8n 대체(8장) |
| 파일저장 | **Supabase Storage** (또는 R2) | 렌더된 카드 PNG → 공개 URL(발행 API 입력) |
| 토큰 암호화 | **libsodium / KMS + 암호화 컬럼** | 인스타 토큰 평문 금지(SPEC 2.2) |
| 생성(LLM) | **Anthropic / OpenAI / Google SDK** | 벤치 결과로 작업별 라우팅 |
| 렌더 | **현행 sharp+SVG 유지 → 필요시 Playwright** | 이미 동작. 디자인 복잡해지면 HTML/CSS+헤드리스로 승급 |
| 호스팅 | **Vercel(프론트) + Railway/Render(워커)** | 워커는 Playwright·큐 돌릴 상시 서버 필요 |
| 모니터링 | **Sentry + 로그** | 발행 실패·웹훅 누락 추적(운영 필수) |

> **렌더 결정**: PoC의 `sharp`+SVG가 이미 PNG를 잘 뽑는다. MVP는 이걸 그대로 쓰고, 디자인 자유도가 필요해지면 Playwright(HTML/CSS 스크린샷)로 갈아탄다. AI 이미지 생성은 비권장(기술설계 4.1) — 아티팩트→고위험 검수 폭증.

## 8. 오케스트레이션 결정: n8n vs 코드(BullMQ)

**결론: 핵심 경로는 코드(BullMQ+cron). n8n은 선택적 보조.** (기술설계 ⑦장 재확정)

| 기준 | n8n | 코드(BullMQ) |
|---|---|---|
| 버전관리·테스트 | 약함 | ✅ 강함 |
| 에러추적/재시도 제어 | 제한적 | ✅ 세밀(Sentry 연동) |
| 토큰·검수상태 등 민감 상태 | 보안표면↑ | ✅ 안전 |
| 별도 시스템 운영부담 | +1 시스템 | 백엔드에 큐만 추가 |
| 빠른 1회성 검증 | ✅ 쉬움 | 코드 필요 |

→ **n8n은 "발행 흐름 빠른 스파이크"나 "내부 알림 자동화"에만 선택 사용.** 프로덕션 backbone은 코드. (이미 발행을 코드로 성공시켰으므로 스파이크 단계도 사실상 통과)

## 9. 개발 계획 (단계별)

> 원칙: **돈·앱리뷰가 드는 것 전에 코드로 닫을 수 있는 것부터.** 각 Phase는 검증 가능한 산출물로 끝난다.

### Phase 1 — LLM 벤치마크 (지금 바로, 키만 있으면)
- `poc/llm-bench/` 하니스 작성: 3 공급자 × 3 시나리오 × N회 → `results.csv` + 결과 덱 렌더
- **산출물**: 시간·비용·품질 표, "카피=모델X / 판단=모델Y" 라우팅 확정
- **의존**: Anthropic/OpenAI/Google API 키 (없으면 mock으로 구조만)

### Phase 2 — 생성 파이프라인 코드화
- 컨셉(주제·타깃·톤) → 전략 → 슬라이드 카피(JSON) → 자가점검/리스크 → `render-deck` 입력
- Phase 1에서 고른 모델로 `generateDeck(concept)` 함수 1개로 묶음
- **산출물**: 로컬에서 컨셉만 주면 검수 가능한 덱 JSON+PNG가 나옴 (입구→렌더 완전 연결)

### Phase 3 — 영속화 기반 (DB·스토리지·OAuth)

- **산출물**: 회원이 인스타 연결하고 컨셉을 DB에 잠그는 흐름

### Phase 4 — 검수 UI (차별화 핵심)
