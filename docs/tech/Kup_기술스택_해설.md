## 1. TypeScript — "오타·버그를 미리 잡아주는 자바스크립트"

**무엇인가**
- 웹은 기본적으로 **JavaScript(JS)** 라는 언어로 돌아간다. 브라우저가 알아듣는 유일한 언어.
- **TypeScript(TS)** 는 JS에 **"타입(type)"** 을 얹은 것이다. 타입 = "이 값은 숫자다 / 글자다 / 이런 모양의 객체다"라고 미리 약속하는 것.
- 코드를 돌리기 전에(빌드 시점에) 타입이 안 맞으면 **에러로 미리 알려준다.** 실행하면 다시 평범한 JS로 변환돼 동작한다.

**왜 우리가 쓰나**
- 비유: JS는 "맞춤법 검사 없는 워드", TS는 "빨간 줄 그어주는 워드". 우리처럼 **개발자 없이 바이브코딩**하면 사소한 실수(필드 이름 오타, 빠뜨린 값)가 런타임에 터지는데, TS는 그걸 **타이핑하는 순간** 잡아준다.
- 우리 데이터 계약이 명확하다([생성파이프라인 설계](Kup_생성파이프라인_설계.md)의 `Concept`/`Deck` 스키마). 이걸 TS 타입으로 박아두면 **deck 만들 때 필드 빠뜨리면 즉시 에러** → 검수·발행까지 가서 깨지는 사고를 막는다.
- **프론트·백엔드·LLM SDK가 전부 JS/TS 한 언어** → 3인 팀이 언어 하나만 알면 됨(인력 효율, 개발계획 §7의 핵심 이유).

**대안과 비교**
| | TypeScript | 그냥 JavaScript | Python |
|---|---|---|---|
| 실수 사전 차단 | ✅ 강함 | ❌ 약함 | 보통 |
| 프론트와 통일 | ✅ | ✅ | ❌ (언어 분리) |
| AI/LLM 생태계 | 좋음 | 좋음 | ✅ 최고 |
→ 우리는 "프론트+백 통일 + 실수 차단"이 핵심이라 **TS**. (LLM 호출은 어차피 API라 언어 무관)

## 2. Next.js + Tailwind — "화면 만드는 틀 + 디자인 빠르게 입히는 도구"

### 2.1 Next.js — "프론트와 백엔드를 한 프로젝트에"

**무엇인가**
- **React**: 화면(UI)을 "컴포넌트"라는 조각으로 조립하는 라이브러리. 페이스북이 만든, 가장 많이 쓰는 프론트 도구.
- **Next.js**: React를 **실제 제품으로 만들 때 필요한 것들(라우팅·서버 렌더링·API)을 다 묶은 프레임워크**. React가 "엔진"이면 Next.js는 "완성차".
- 결정적 특징: **API Route** — 같은 프로젝트 안에서 "백엔드 API"도 만들 수 있다. 즉 `/검수화면`(프론트)과 `/api/decks`(백엔드)가 한 코드베이스.

**왜 우리가 쓰나**
- 우리가 만들 화면: **검수 UI(시안 보고 5초 승인)**, **대시보드(팔로워 순증 등)**, 온보딩(컨셉 잠금). 전형적인 React/Next.js 영역.
- 프론트와 백엔드 API를 **한 프로젝트**에서 → 작은 팀이 빠르게. (단, 무거운 백그라운드 작업=발행 워커는 분리 — §5, §10에서 설명)
- Vercel(만든 회사)에 **그냥 올리면 배포 끝**(§10 호스팅).

### 2.2 Tailwind — "미리 만든 디자인 조각으로 빨리 꾸미기"

**무엇인가**
- CSS(화면 꾸미는 언어)를 매번 새로 쓰는 대신, `text-lg`(큰 글자) `p-4`(안쪽 여백) `rounded-xl`(둥근 모서리) 같은 **미리 정의된 클래스**를 HTML에 붙여 디자인하는 도구.

**왜 우리가 쓰나**
- 디자이너 없이 **빠르게 일관된 UI**를 찍어낼 수 있다. 검수 UI·대시보드를 빠르게 만드는 게 목표라 적합.
- ⚠️ **주의**: Tailwind는 **웹 화면(검수 UI)** 꾸미는 것이고, **카드뉴스 이미지(인스타에 올라가는 PNG)** 와는 **무관**하다. 카드뉴스는 별도 렌더 파이프라인(§8). 이 둘을 헷갈리지 말 것.

**대안**: Bootstrap(구식), 직접 CSS(느림), MUI(무겁고 디자인 획일). → 속도·자유도 균형으로 Tailwind.

## 3. DB: Supabase + Postgres — "데이터를 보관하는 곳"

### 3.1 Postgres — "표(table)들의 집합, 가장 믿음직한 관계형 DB"

**무엇인가**
- **데이터베이스(DB)**: 회원·채널·시안·게시물 같은 정보를 **영구 보관**하는 곳. (코드가 꺼져도 남음)
- **Postgres(PostgreSQL)**: 가장 널리 쓰이는 **관계형 DB**. "관계형" = 데이터를 **엑셀 같은 표(table)** 로 저장하고, 표끼리 **연결(관계)** 한다.
  - 예: `users` 표의 회원 1명이 → `channels` 표의 인스타 계정 여러 개를 가짐 → 각 채널이 `drafts`(시안) 여러 개를 가짐. 이 "1명-여러개" 연결이 관계형의 핵심.

### 3.2 Supabase — "Postgres + 인증 + 파일저장을 한 번에 주는 관리형 서비스"

**무엇인가**
- Postgres를 **직접 서버에 깔고 운영(백업·보안·확장)** 하는 건 번거롭다. **Supabase**는 그 Postgres를 **클라우드에서 관리해주고**, 거기에 **인증(로그인)·파일저장·자동 API**까지 얹어주는 서비스. ("Firebase의 오픈소스/Postgres 버전"이라 불림)
- 즉 Supabase 안에 진짜 Postgres가 들어있다. **벤더 종속이 약하다** — 나중에 떠나도 표준 Postgres라 그대로 옮길 수 있음.

**왜 우리가 쓰나**
- 3인 팀이 DB 운영에 시간을 쓸 수 없다 → 관리형으로 **운영 부담 제거**.
- **인증·스토리지가 한 세트** → 따로 붙일 서비스가 줄어 속도↑ (§4, §6).

### 3.3 "설계는 어떻게 하나" — 데이터 모델링 3단계

DB 설계 = **"어떤 표를 만들고, 표끼리 어떻게 연결하고, 각 표에 무슨 칸(컬럼)을 둘지"** 정하는 일. 우리는 이미 [기술설계 §8](Kup_기술설계.md)에 초안이 있다.

**1) 표(엔티티) 뽑기** — "저장해야 하는 명사"를 찾는다:
```
users(회원) · channels(연결된 인스타) · ig_tokens(암호화 토큰) ·
channel_config(잠긴 컨셉) · drafts(시안) · reviews(검수기록) ·
posts(발행물) · lead_rules(리드규칙) · leads(리드) · dm_logs(DM기록) ·
events(제품이벤트) · metrics_daily(일별성과)
```

**2) 관계 잇기** — 표끼리 "누구 소속인지"를 외래키(foreign key)로 연결:
```
users ─1:N─ channels ─1:1─ ig_tokens
                     ─1:1─ channel_config
                     ─1:N─ drafts ─1:N─ reviews
                                  ─1:1─ posts ─1:N─ lead_rules ─1:N─ leads
```
("1:N" = 한 회원이 여러 채널, "1:1" = 한 채널당 토큰 하나)

**3) 컬럼·타입·제약 정하기** — 각 표에 무슨 칸을, 무슨 타입으로:
```sql
-- 예: drafts(시안) 표
create table drafts (
  id          uuid primary key default gen_random_uuid(),
  channel_id  uuid references channels(id),     -- 어느 채널 소속인지
  topic       text not null,
  slides      jsonb not null,                   -- deck의 slides 배열을 통째로
  caption     text,
  hashtags    text[],
  risk_level  text check (risk_level in ('low','high')),
  ai_flags    jsonb default '[]',
  status      text default 'pending',           -- pending/approved/rejected
  created_at  timestamptz default now()
);
```
> 포인트: 우리 [Deck 스키마](Kup_생성파이프라인_설계.md)가 이미 정의돼 있어서, **그 JSON 구조를 그대로 `jsonb` 컬럼**(slides 등)으로 넣거나, 정규화해서 펼친다. MVP는 `jsonb`로 단순하게 시작 권장.
> 실제 작업: SQL로 `create table`을 쓰거나, Supabase 대시보드/마이그레이션 도구로 만든다. **"마이그레이션"** = 표 구조 변경 이력을 코드로 관리하는 것(버전관리처럼).

## 4. Supabase Auth — "우리 서비스 자체 로그인" (인스타 연결과 별개!)

**무엇인가**
- 회원이 **Kup에 가입·로그인**하는 기능(이메일/비번, 구글 로그인 등). Supabase가 제공.

**왜 따로 설명이 필요한가 — 두 개의 로그인을 혼동 금지**
| | **Supabase Auth** | **인스타 OAuth** |
|---|---|---|
| 무엇 | Kup 서비스에 로그인 | 사용자의 **인스타 계정 연결** |
| 목적 | "너 누구야"(우리 회원 식별) | "네 인스타에 우리가 대신 올려도 돼?"(권한 위임) |
| 결과물 | 세션(로그인 상태) | 인스타 **액세스 토큰**(→ 암호화 저장, §7) |
| 시점 | 가입 즉시 | 온보딩 중 "인스타 연결" 버튼 |

→ **Auth = 우리집 현관 열쇠**, **인스타 OAuth = 인스타가 우리한테 준 대리 출입증**. 완전히 다른 것. (기술설계 §6.1)

**왜 Supabase Auth**: DB·스토리지와 한 세트라 따로 붙일 필요 없음. 대안 Clerk(더 예쁜 로그인 UI, 유료)도 가능하나 MVP는 통합이 이득.

## 5. 작업큐 / 예약: BullMQ + Redis — ★ 가장 중요, 자세히

### 5.1 문제부터 — "예약 발행"은 왜 어려운가

사용자가 "이 시안, **내일 아침 8시에** 올려줘"라고 승인했다. 그런데:
- 웹 서버는 **요청이 올 때만 잠깐 일하고 끝난다.** "내일 8시까지 기다렸다가 올려"를 직접 못 한다(서버가 그동안 놀 수도, 재시작될 수도 있음).
- 인스타 발행은 **3단계(컨테이너 생성→처리완료 폴링→게시)** 라 시간이 걸리고, **실패하면 재시도** 해야 한다.
→ 이런 "**나중에 / 오래 걸리는 / 실패하면 다시**" 작업은 **별도 장치**가 필요하다. 그게 작업큐.

### 5.2 용어 셋 — Redis / 작업큐 / 워커

- **Redis**: 초고속 메모리 DB. 여기선 **"할 일 목록(큐)을 보관하는 메모장"** 역할.
- **작업큐(Job Queue)**: "해야 할 일"을 **줄 세워 보관**하는 통. 예약 시각·재시도 횟수 같은 정보를 함께 들고 있음.
- **BullMQ**: Node.js(TS)에서 Redis 기반 작업큐를 쉽게 쓰게 해주는 **라이브러리**.
- **워커(Worker)**: 큐에서 일을 **하나씩 꺼내 실제로 처리**하는 별도 프로그램(상시 켜져 있음).

비유: **식당 주방**.
- 손님(웹 요청)이 주문서를 **주문 꽂이(Redis 큐)** 에 꽂는다 →
- **요리사(워커)** 가 꽂이에서 하나씩 빼서 요리(발행) →
- 타면(실패) **다시 시도**, 계속 실패하면 "이 주문 문제 있음" 알림.

### 5.3 우리 발행 흐름에 대입

```
[검수 UI] 사용자가 승인 + "내일 8시 예약"
   │  → BullMQ에 "발행 잡"을 delay(내일 8시)로 등록
   ▼
[Redis 큐]  잡이 8시까지 대기
   ▼ (8시 도달)
[발행 워커]  잡을 꺼냄 →
   1) AI 라벨 부착·캡션 최종화
   2) 인스타 컨테이너 생성(슬라이드 이미지 URL들)
   3) FINISHED 될 때까지 상태 폴링
   4) 게시(publish) → 게시물 ID 받음 → posts 테이블 저장
   실패 시 → 백오프 후 재시도(3회), 계속 실패면 사용자 알림 + 검수피드 표시
```

### 5.4 왜 BullMQ인가 (cron만으론 부족한 이유)

- **cron**: "매일 8시에 실행" 같은 **반복 스케줄**. 우리도 **주기 작업(성과 수집)** 엔 cron 씀.
- 하지만 cron은 "**이 시안 하나를 정확히 이 시각에, 실패하면 3번 재시도하고, 진행상태를 추적**"을 못 한다. → 그건 **작업큐(BullMQ)** 의 일.
- 정리: **cron = 반복 알람**, **BullMQ = 개별 작업의 예약·재시도·추적**. 우리는 **둘 다** 쓴다(cron으로 주기 잡 투입 → BullMQ가 처리).

### 5.5 왜 이게 "핵심 비즈니스 로직"이라 코드여야 하나
발행·재시도·웹훅 처리는 **민감한 토큰·검수 상태**를 다루고, **버전관리·테스트·에러추적(Sentry)** 이 필수다. → 비주얼 도구(n8n)가 아니라 **코드(BullMQ)** 로 둔다. (이 결론의 상세 근거 = [n8n 결정 문서](Kup_n8n_결정.md))

## 6. 파일 저장: Supabase Storage vs Cloudflare R2 — 비교

### 6.1 왜 파일 저장이 필요한가
인스타 발행 API는 **"이미지 파일"을 직접 안 받는다.** 인터넷에 공개된 **이미지 URL**을 받는다. 그래서 우리가 만든 카드 PNG를 **어딘가 올려서 공개 URL을 만들어야** 한다. 그 "어딘가" = 오브젝트 스토리지.

### 6.2 둘 비교

| 기준 | **Supabase Storage** | **Cloudflare R2** |
|---|---|---|
| 정체 | Supabase에 내장된 파일 저장소 | 클라우드플레어의 S3 호환 스토리지 |
| 장점 | DB·인증과 **한 세트**, 권한 연동 쉬움, 설정 간단 | **데이터 송출(egress) 비용 0**, 대용량·고트래픽에 저렴, CDN 친화 |
| 단점 | 트래픽 많아지면 비용·한도 부담 | 별도 계정·설정, 학습 약간 |
| 우리 용도 적합도 | **MVP에 최적**(이미 Supabase 씀 → 추가 0) | **확장기에 최적**(이미지 조회 폭증 시) |

→ **결론: MVP는 Supabase Storage로 시작**(추가 서비스 없음, 가장 빠름). 카드 이미지 **조회 트래픽이 커지면 R2로 이전**(egress 무료라 비용 급감). 둘 다 S3 호환이라 이전 부담 작음.

## 7. 토큰 암호화 — "남의 인스타 출입증을 금고에 넣기"

**토큰이 뭔가**
- 사용자가 인스타를 연결하면, 인스타가 우리한테 **액세스 토큰**(60일짜리 장기 토큰)을 준다. 이건 **"이 계정에 글 올리고 DM 보낼 수 있는 대리 출입증"** 이다.

**왜 암호화가 필수인가**
- 토큰이 유출되면 **공격자가 사용자의 인스타에 마음대로 발행·DM** 할 수 있다 = 심각한 보안 사고, 신뢰 붕괴, 법적 책임.
- 그래서 DB에 **평문(그냥 글자)으로 저장 절대 금지**(SPEC 2.2). 반드시 **암호화**해서 저장 → DB가 통째로 털려도 토큰은 못 읽게.

**어떻게**
- **암호화** = 비밀 키로 토큰을 알아볼 수 없는 형태로 바꿔 저장, 쓸 때만 그 키로 복호화.
- 방법: **libsodium**(암호화 라이브러리)로 앱에서 암·복호화하거나, **KMS**(클라우드 키 관리 서비스)에 키를 맡기고 **DB 암호화 컬럼**에 저장.
- 비번은 **아예 안 받는다**(OAuth라 토큰만 받음). → 받을 게 없으니 샐 것도 없음.

## 8. 렌더 방식 — "텍스트(카피)를 실제 카드 이미지(PNG)로"

### 8.1 렌더가 뭔가
LLM은 **글자(카피)** 만 만든다. 그걸 **브랜드 컬러·레이아웃이 입혀진 정사각 이미지(PNG)** 로 바꾸는 작업 = 렌더. 이 PNG가 인스타에 올라간다.

### 8.2 현행 방식 (PoC, sharp + SVG)
- `poc/render-deck.js`: deck JSON을 받아 → **SVG**(코드로 그리는 벡터 그래픽)로 카드를 그리고 → **sharp**(이미지 라이브러리)로 PNG 변환.
- 이미 동작한다: 그라데이션 배경·진행 점·번호·푸터·AI라벨까지. (1080×1350px)
- **장점**: 외부 비용 0, 완전 코드 제어. **단점**: 디자인이 복잡해지면 SVG를 손으로 그리기 빡셈, 글자수 제약이 빡빡함([설계 §1.3](Kup_생성파이프라인_설계.md)).
- ⚠️ **"다 비슷해 보임" 문제의 원인 = 디자인 다양성은 프롬프트가 아니라 "템플릿 종류"에서 나온다.** 현행은 레이아웃 템플릿이 사실상 1개라, 채널이 달라도 색만 바뀌고 뼈대가 같아 비슷해 보인다. **프롬프트(Claude)는 글(deck)만 만들고 디자인은 1도 안 건드린다**(§9 원칙)는 점을 기억할 것 → 해법은 프롬프트 튜닝이 아니라 **템플릿 변형 추가**.
- ✅ **실증(테스트 A)**: `poc/render-variants.js`로 같은 deck을 **뼈대가 다른 레이아웃 4종**(A 그라데이션 / B 에디토리얼 / C 센터미니멀 / D 밴드대비)으로 렌더 → 글은 그대로인데 4가지 인상이 나옴(`out/variants/compare_slide*.png`). **SVG만으로도 "다 비슷함" 문제는 1차 해결됨.** 실서비스에선 deck에 `layout: "editorial"` 같은 **선택값**만 AI가 고르게 하면(=그리는 게 아니라 고르기) 원칙을 안 깨고 다양성 확보.

### 8.3 승급 경로 (Playwright = HTML/CSS 렌더) — "준비된 비상구"
- 디자인 자유도가 필요해지면: 카드를 **웹페이지(HTML/CSS)** 로 만들고 → **Playwright**(보이지 않는 브라우저)로 **스크린샷** → PNG.
- **장점**: 웹 디자인하듯 자유롭게(글자 자동 줄임·`text-wrap:balance`, 그림자, 블러, flex 정렬, 폰트), 글자수 제약 완화. 무엇보다 **만들기가 쉬워짐**(CSS가 정렬·여백·줄바꿈 자동, SVG는 좌표 손계산).
- **단점/비용**: 브라우저를 돌릴 **상시 서버** 필요 + Chromium 바이너리(~93MB)·렌더 시 메모리 수백 MB → 워커 사양 한 단계 ↑(대략 월 $5~7 → $10~25).
- ⭐ **상시 서버는 "새로 만드는 것"이 아니다.** 예약 발행·웹훅·큐 처리 때문에 **워커는 이미 상시 서버(Railway/Render, §10.3)** 로 돌고 있다. Playwright는 그 워커에 **브라우저 기능을 얹는 것**일 뿐 — 서버가 늘지 않고 사양만 올라간다. ("상시 서버 안 씀"은 Vercel 웹앱 쪽 얘기. 워커는 원래부터 상시.)
- ✅ **실증(테스트 B)**: `poc/render-playwright.js`로 같은 deck을 HTML/CSS 렌더 → SVG 대비 **다층 그라데이션·블러 발광·그림자·자동 균형 줄바꿈**으로 눈에 띄게 고급스러움(`out/playwright/vs_slide*.png`). **퀄리티 우위는 분명하나, "지금 갈아탈 만큼"은 아님**(테스트 A로 1차 해결되므로).
- 🔧 **준비 상태(갈아탈 때 바로 쓰게)**: `poc/render-playwright.js` 뼈대 완성·동작 확인됨. deck 구조·브랜드 컬러·cover/body/outro 3종 모두 SVG 렌더러와 **동일 입력 호환**. 전환 시 할 일 = ① 워커 호스팅에 `npx playwright install chromium` ② 렌더 호출부를 `render-deck.js`→`render-playwright.js`로 스왑 ③ 워커 사양 상향. (deck 스키마 변경 불필요.)

### 8.4 결론 (개발계획 §7 재확인 + 상세)
> **MVP는 현행 sharp+SVG로 가되, "템플릿 다양화"로 다양성을 확보한다.** 비용 0, 이미 PNG 잘 나옴.
> **Playwright는 "준비된 비상구"로 코드·뼈대를 미리 갖춰둔다**(`render-playwright.js`). **갈아타는 트리거**: ① SVG 템플릿으로 도저히 안 나오는 디자인 요구가 생기거나 ② 사진/리치 타이포가 카드의 핵심이 될 때. 그 순간 빠르게 스왑(§8.3 준비 상태 참고).
> **AI 이미지 생성(텍스트→그림)은 비권장**(기술설계 §4.1): 손가락·글자 왜곡 등 아티팩트 → **고위험 검수 폭증** → "신뢰·5초 검수" 포지션과 정면충돌. 사진이 필요하면 **사용자 업로드 자산(B모드)** 또는 스톡.
> 즉 **"글=AI, 디자인=템플릿"** 으로 책임을 분리하는 게 품질 안정의 핵심. (A/B 테스트로 이 판단 실증됨 — 2026-06-24)

## 10. 호스팅: Vercel(프론트) + Railway/Render(워커) — "어디에 올려 돌리나"

### 10.1 왜 둘로 나누나 — 두 종류의 일이 성격이 다르다

| | **웹앱(프론트+API)** | **워커(발행·렌더)** |
|---|---|---|
| 성격 | 요청 오면 **잠깐** 응답하고 끝 | **상시 켜져** 큐를 지켜보며 일함 |
| 예 | 검수 화면 열기, 대시보드 | 예약 발행, 이미지 렌더, 웹훅 처리 |
| 필요 환경 | 서버리스(자동 확장, 요청당 과금) | **상시 실행 서버**(큐·Playwright 돌릴) |
| 적합 호스팅 | **Vercel** | **Railway / Render / Fly** |

### 10.2 Vercel — "Next.js를 그냥 올리면 배포되는 곳"
- Next.js 만든 회사. 코드 푸시하면 **자동 빌드·배포·전세계 CDN**. 프론트+API에 최적.
- 단, Vercel은 **요청-응답형(서버리스)** 이라 "**몇 시간 뒤 발행**" 같은 **장시간·상시 작업엔 부적합**. → 그래서 워커를 분리.

### 10.3 Railway / Render — "상시 켜진 서버(워커용)"
- **BullMQ 워커**(§5)와 **Playwright 렌더**(§8.3)는 **계속 켜져 있어야** 한다(큐 감시, 브라우저 구동). 이런 "상시 프로세스"를 쉽게 올리는 호스팅이 Railway·Render·Fly.
- 셋 다 비슷. **Railway**(설정 가장 쉬움) 또는 **Render**(무료 티어·문서 친절) 중 택1. MVP는 어느 쪽이든.

### 10.4 그림으로
```
[사용자 브라우저]
     │
     ▼
[Vercel]  Next.js 웹앱+API   ── 검수/대시보드/온보딩, "발행 잡 등록"만 함
     │  (잡을 Redis 큐에 넣음)
     ▼
[Upstash Redis] 작업큐
     ▲
     │ (큐 감시·처리)
[Railway/Render]  발행 워커 + 렌더   ── 실제 인스타 발행·이미지 생성·웹훅
     │
     ▼
[Meta 인스타 API]
```

|---|---|---|
