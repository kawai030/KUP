# Kup — 생성 파이프라인 & 프롬프트 상세 설계

> **문서 성격**: "어떤 카드뉴스를, 어떤 데이터 계약과 프롬프트로 만들 것인가"를 코드로 옮기기 직전까지 확정하는 설계 문서.
> **왜 지금**: LLM 벤치마크(모델 선정)는 *이 설계가 먼저 고정돼야* 의미가 있다 — 세 모델에 **동일한 입력·동일한 프롬프트·동일한 출력 스키마**를 꽂아야 공정하기 때문. 즉 이 문서가 벤치마크의 입력 사양이다.
> **기준일**: 2026-06-23 · **선행**: [SPEC](../product/Kup_SPEC.md) · [기술설계](Kup_기술설계.md) §6.2 · [기술스택_개발계획](Kup_기술스택_개발계획.md) Phase 1~2
> **근거 코드**: `examples/concepts/*.json`(잠긴 컨셉 예시), 검증 PoC(비공개)의 deck 예시, `lib/render/deck-renderer.ts`(렌더 제약의 출처 — PoC SVG 렌더를 이식)

---

## 0. 한눈에 — 파이프라인 5단계

```
[잠긴 컨셉]  ← 채널 1회 설정, 이후 모든 생성의 고정 제약 (= 시스템 프롬프트 = 캐싱 대상)
  persona / tone / pillars[] / cadence / visual{}
     │
     ▼
① 주제 결정     pillars 안에서 후보 주제 N개 제안 → 1개 선택(사용자 or 자동)
     ▼
② 전략 설계     선택 주제 → 후킹 앵글 + 슬라이드 구성(장수·슬라이드별 역할)
     ▼
③ 카피 생성     슬라이드별 텍스트 + 캡션 + 해시태그  → **deck JSON** (구조화 출력)
     ▼
④ 자가 점검     미검증 사실·규제 표현·실존 대상 → ai_flags[] + risk_level
     ▼
⑤ 렌더         deck JSON → 글자수 제약 검증 → PNG (lib/render/deck-renderer.ts)
```

핵심 원칙: **①②④는 "판단"이 필요한 단계(상위 티어 후보), ③은 "제약 안에서 짧은 텍스트 생성"(저가 티어 후보).** 단계를 나누는 이유가 곧 모델 라우팅의 근거다(§5). 단, MVP 1차는 **단일 호출(②③④를 한 번에)**로 시작해 지연·비용을 줄이고, 벤치 결과로 분리 여부를 정한다(§6).

## 1. 데이터 계약 (스키마) — "정형화된 카드뉴스"

PoC의 수기 JSON을 **타입·필수·제약이 있는 계약**으로 승격한다. 이게 generateDeck()의 출력이자 렌더러의 입력이며 벤치마크의 채점 대상이다.

### 1.1 Concept (채널이 1회 잠그는 것 — 변하지 않음)

```jsonc
{
  "id": "playlist",                  // string, 채널 식별
  "account": "@em0rm0re",            // string, 표시용 핸들
  "persona": "취향 좋은 새벽 감성 음악 큐레이터",  // string, 1문장
  "tone": "잔잔하고 담백하게, 과한 수식어 자제",   // string, 1문장 (카피 스타일 제약)
  "pillars": ["상황별 플레이리스트", "장르 입문", "아티스트 큐레이션"], // string[], 3~5개
  "cadence": "주 2회",               // string
  "visual": {                        // 렌더 브랜드 (생성과 무관, 렌더 단계 입력)
    "primary": "#14122B", "primary2": "#2E2357",
    "accent": "#9D8DF1", "light": "#F3F1FA",
    "font": "Apple SD Gothic Neo, AppleGothic, sans-serif"
  }
}
```

→ persona·tone·pillars는 **모든 생성 호출의 시스템 프롬프트에 그대로 들어간다**(채널마다 고정 → 프롬프트 캐싱으로 입력비 ~90% 절감).

### 1.2 Deck (LLM이 생성하는 것 — 매 콘텐츠)

```jsonc
{
  "conceptId": "playlist",           // string, 어떤 컨셉으로 생성됐는지
  "topic": "잠 안 오는 새벽에 듣기 좋은 플레이리스트", // string, ① 단계 산출
  "strategy": "공감형 후킹 → 무드별 3장 → 댓글 리드마그넷", // string, ② 단계 산출(검수자용 설명)
  "slides": [ /* §1.3 — 5장 권장(3~10 허용) */ ],
  "caption": "…",                    // string, 인스타 캡션 (≤ 2200자, 실질 권장 ≤ 300자)
  "hashtags": ["#새벽감성", "…"],     // string[], 5~10개, 각 '#' 시작·공백 없음
  "ai_flags": ["실존 곡명 언급 시 검수 필요"], // string[], ④ 단계 산출 (없으면 [])
  "risk_level": "low"                // "low" | "high", ④ 단계 산출
}
```

### 1.3 Slide — **렌더러가 강제하는 글자수 제약**(가장 중요)

`render-deck.js`의 `wrap(text, maxChars)`가 줄당 글자수를 잘라준다. **안 깨지게 하려면 LLM이 줄 수까지 고려해 생성**해야 한다. 아래 "권장 최대"는 카드가 시각적으로 안정적인 상한(= 프롬프트에 글자수 룰로 박는다).

| kind | 필드 | 줄당 wrap | 권장 줄수 | **권장 최대 글자(공백 포함)** | 비고 |
|---|---|---|---|---|---|
| **cover** | `kicker` | pill(1줄) | 1 | **10** | 카테고리 라벨 |
| | `title` | 9자 | 2~3 | **24** (`\n`로 줄나눔 가능) | 가장 큰 글자 |
| | `sub` | 20자 | 1~2 | **34** | 부제 |
| **body** | `index` | — | 1 | `"01"`,`"02"`… 2자리 | 슬라이드 번호 |
| | `head` | 13자 | 1~2 | **24** | 소제목 |
| | `body` | 17자 | 2~3 | **48** (`\n` 가능) | 본문 |
| **outro** | `title` | 12자 | 1~2 | **22** | CTA 후킹 |
| | `sub` | 20자 | 2 | **40** | 행동 안내(댓글 키워드) |
| | `cta` | pill(1줄) | 1 | **14** | 예 `💬 댓글: 플리` |

> 슬라이드 구성 규칙: **1장 cover + N장 body + 1장 outro**. body는 1~8장(전체 3~10장). outro의 `cta`는 리드마그넷 키워드와 일치해야 함(§4, 웹훅 매칭 키).
> 이 표는 **현행 sharp+SVG 렌더 기준**. 추후 Playwright(HTML/CSS) 렌더로 가면 자동 축소(auto-fit)가 쉬워져 제약이 완화될 수 있다 — 그땐 이 표를 갱신한다.

## 2. 단계별 프롬프트 설계

> 표기: `{{변수}}`는 런타임 주입. 시스템 프롬프트(고정/캐싱) vs 유저 프롬프트(매번 변동)를 분리한다.

### 2.0 시스템 프롬프트 (모든 단계 공통 · 캐싱 대상)

```
너는 인스타그램 카드뉴스 카피라이터다. 아래 "채널 컨셉"을 절대 규칙으로 따른다.

[채널 컨셉]
- 페르소나: {{persona}}
- 톤: {{tone}}
- 콘텐츠 기둥(이 안에서만 주제): {{pillars}}

[불변 규칙]
1. 위 톤·페르소나를 모든 문장에 반영한다. 과장·클릭베이트·이모지 남발 금지.
2. 사실로 단정할 수 없는 것(수치·효능·실존 인물/매장/곡명)은 만들어내지 않는다.
   불확실하면 일반화하거나 ai_flags에 기록한다.
3. 광고/규제 표현(금융=수익 보장·투자 권유, 의료=치료·효능 단정)을 쓰지 않는다.
4. 출력은 지정된 JSON 스키마를 정확히 따른다. 글자수 제약을 반드시 지킨다.
```

→ 이 블록이 채널마다 고정이라 **prompt caching** 적용 1순위. (Claude `cache_control`, OpenAI 자동 캐시, Gemini `cachedContent` — 3사 모두 지원)

### 2.1 ① 주제 결정 프롬프트 (유저)

```
[작업] 이 채널의 다음 카드뉴스 주제를 {{n=5}}개 제안하라.
- 각 주제는 pillars 중 하나에 속해야 한다.
- 저장·공유를 부르는 "정보성 + 공감" 앵글을 우선한다.
- (선택) 참고 신호: {{recent_topics 최근 다룬 주제들}} 과 겹치지 않게.

[출력 JSON] { "topics": [{ "title": string, "pillar": string, "angle": string }] }
```
사용자가 1개 고르거나, 자동 모드면 첫 번째 채택. (MVP: 인스타 외부 트렌드 데이터 없음 → 컨셉 기반 생성. 추후 자사 성과로 가중 — 기술설계 §6.2)

### 2.2 ② 전략 설계 프롬프트 (유저)

```
[작업] 선택 주제 "{{topic}}"로 카드뉴스 1세트의 구성을 설계하라.
- 슬라이드 장수 결정(cover 1 + body 3~6 + outro 1 권장).
- 후킹 앵글(cover에서 멈추게 할 1문장 컨셉)을 정한다.
- 각 body 슬라이드가 무엇을 말할지 한 줄 요약(아직 카피 아님).
- 마지막 outro의 리드마그넷 후킹(무엇을 댓글로 받게 할지)을 정한다.

[출력 JSON] {
  "strategy": string,            // 검수자용 한 줄 설명
  "hook": string,
  "slidePlan": [{ "kind": "cover"|"body"|"outro", "purpose": string }],
  "leadKeyword": string          // outro cta에 쓸 키워드 (예: "플리")
}
```

### 2.3 ③ 카피 생성 프롬프트 (유저) — **핵심, 구조화 출력**

```
[작업] 아래 구성(slidePlan)에 맞춰 실제 카드뉴스 카피를 작성하라.
주제: {{topic}} / 앵글: {{hook}} / 구성: {{slidePlan}} / 리드키워드: {{leadKeyword}}

[글자수 제약 — 초과 금지]
- cover: kicker≤10, title≤24(줄바꿈 \n 가능), sub≤34
- body : head≤24, body≤48(\n 가능), index는 "01"부터 2자리
- outro: title≤22, sub≤40, cta≤14 (cta에 리드키워드 포함, 예 "💬 댓글: {{leadKeyword}}")
- caption ≤ 300자, hashtags 5~10개(각 '#'으로 시작, 공백 없음)

[출력 JSON] deck 스키마(§1.2)의 topic·strategy·slides·caption·hashtags 까지.
ai_flags·risk_level 은 비워서 출력(다음 단계에서 채움).
```

### 2.4 ④ 자가 점검 프롬프트 (유저)

```
[작업] 아래 deck을 검수자 입장에서 점검하라. 너의 생성물을 비판적으로 본다.
{{deck JSON}}

[점검 항목]
- 사실로 검증이 필요한 주장(수치·효능·실존 인물/매장/곡명·날짜)
- 규제·광고 표현(금융 권유, 의료 효능, 과장 보장)
- 톤/페르소나 일탈

[판정 규칙]
- 위 위험이 하나라도 있으면 risk_level="high", 해당 지점을 ai_flags에 구체적으로.
- 전혀 없으면 risk_level="low", ai_flags=[].

[출력 JSON] { "ai_flags": string[], "risk_level": "low"|"high" }
```

> 이 단계 품질이 SPEC의 "무수정 승인율 60~85%"·"5초 검수"를 좌우한다. **자가점검 정확도가 곧 모델 선택의 결정 변수**(싼 모델이라도 위험을 놓치면 검수 시간이 늘어 진다 — [개발계획 §4](Kup_기술스택_개발계획.md) 루브릭).

## 3. 구조화 출력(JSON) — 3사 공통 전략

벤치마크 공정성을 위해 셋 다 **강제 JSON 스키마**로 받는다(파싱 실패=구조 점수 0):

| 공급자 | 방식 |
|---|---|
| Anthropic | tool use(입력 스키마를 tool로) 또는 프롬프트+JSON 검증 재시도 |
| OpenAI | `response_format: json_schema`(strict) |
| Google | `responseMimeType: application/json` + `responseSchema` |

공통: 출력 후 **Zod 스키마로 검증** → 실패 시 1회 재생성(repair). 글자수 초과는 검증에서 잡아 재생성 트리거.

## 4. 리드마그넷 일관성 (생성 ↔ 발행 ↔ 웹훅)

`outro.cta`의 키워드(`leadKeyword`)는 단순 텍스트가 아니라 **운영 키**다:
- 생성: outro에 "댓글 '{{leadKeyword}}'" 노출
- 발행 후: `lead_rules`에 `{ keyword: leadKeyword, dm_payload }` 저장 (기술설계 §8)
- 웹훅: 댓글에 그 키워드 → Private Reply DM (기술설계 §6.5)

→ 생성 단계에서 `leadKeyword`를 deck에 명시적으로 보존해야 발행/웹훅이 자동 연결된다. (현재 PoC 덱은 outro.cta 안에만 있음 → **deck 최상위에 `leadKeyword` 필드 추가 권장**.)

## 5. 모델 라우팅 (벤치마크로 확정할 가설)

| 단계 | 작업 성격 | 1차 가설 티어 | 비고 |
|---|---|---|---|
| ① 주제 | 발산, 가벼움 | 저가 (Haiku/Flash/GPT-5) | 품질 영향 작음 |
| ② 전략 | 약한 판단 | 중가 (Sonnet/GPT-5.4/Pro) | 후킹 품질이 전체 좌우 |
| ③ 카피 | 제약 내 생성 | **저가** | 변동비 핵심 → 최저가 후보 |
| ④ 자가점검 | 판단·정확성 | **중·상위** | 놓치면 검수 붕괴 |

> 이 표는 가설. 벤치마크가 "③도 저가로 충분한가", "④에 추론(thinking) 필요한가"를 숫자로 답한다. → [개발계획 §5 벤치 하니스]가 검증 도구.

## 6. 함수 인터페이스 (Phase 2 구현 목표)

```ts
// 단일 진입점. MVP 1차는 내부적으로 ②③④를 1~2회 호출로 묶는다.
async function generateDeck(
  concept: Concept,
  opts?: { topic?: string; slideCount?: number; variants?: number }
): Promise<{ deck: Deck; usage: TokenUsage; timings: Timings }>;
```
- `topic` 미지정 → ① 주제 제안 후 첫 번째 채택(또는 사용자 선택 콜백).
- `variants` → 카피만 N개 재생성(검수 UI 변형안). **비용 N배** → 정책 미정(나중 결정).
- `usage`/`timings` 반환 → 그대로 벤치마크 계측에 사용(같은 함수로 3사 측정).

산출물(Phase 2 완료 기준): **컨셉만 주면 검수 가능한 deck JSON + PNG가 로컬에서 나온다**(입구→렌더 완전 연결). [개발계획 Phase 2](Kup_기술스택_개발계획.md)와 동일.
