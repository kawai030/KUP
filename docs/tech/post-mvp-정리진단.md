# Post-MVP 정리 진단 리포트

**목적** — MVP 발표 이후, 다음 개발을 시작하기 전에 코드 상태를 점검하고 "정리할 목록"을 만든다.
이 문서는 **진단만** 한다. 코드는 고치지 않는다. 실제 수정은 이 목록을 함께 검토한 뒤 항목별 작은 PR로 진행한다.

## 베이스라인 (정리 전 상태)

정리를 시작하기 전, 현재 상태가 건강한지 확인했다. 3종 전부 통과다.

| 검사 | 결과 | 비고 |
|---|---|---|
| `npm run typecheck` | 통과 | `.next` 캐시가 낡으면 가짜 에러가 뜬다 → 캐시 삭제 후 재검사로 확인 |
| `npm run lint` | 통과 (경고만) | 에러 0, 경고 8건 (F-4) · `next lint` 자체가 deprecated |
| `npm run build` | 통과 | 프로덕션 빌드 정상 |

여기서부터 정리하면서 이 3종이 계속 통과하는지만 확인하면, 회귀(내가 뭔가 깨뜨림)를 즉시 잡을 수 있다.

## 강점 (이미 잘 되어 있는 것 — 유지)

정리한다고 다 뜯을 필요는 없다. 아래는 이미 현업 수준이다.

- **문서가 풍부하다.** `docs/product`·`docs/tech`·`docs/reference`로 잘 분류돼 있고 설계 SoT가 살아 있다.
- **공유 계약이 명시돼 있다.** `lib/deck-schema.ts`·`lib/concept-schema.ts` 등 "바꾸면 위험한 것"이 CLAUDE.md에 문서화됨.
- **의도된 빈칸이 표시돼 있다.** `TODO(Phase5)` 스텁(`lib/sentry.ts`, `workers/jobs/publish.ts`)은 죽은 코드가 아니라 계획된 자리.
- **폴더 경계가 깔끔하다.** `lib/{generate,llm,render,db,supabase,workspace}` 책임 분리가 명확.
- **CI 게이트가 있다.** typecheck·lint·build 3종이 PR 전 필수로 규칙화됨.

## 발견 항목 (Findings)

### F-1. 테스트 0개 — 안전망 부재

상태: 위험 · 우선순위: 높음

자동 테스트가 하나도 없어서, 기능이 잘 도는지 매번 사람이 손으로 확인해야 한다.
이 상태에서는 리팩토링이나 기능 추가를 할 때 무엇이 깨졌는지 자동으로 잡을 수 없다. 포트폴리오 관점에서도 "테스트를 짤 줄 안다"는 신호가 없다.

**방향** — Vitest(Next+TS와 궁합이 좋다)를 세팅하고, 핵심 순수 로직부터 테스트를 더한다.
가장 가치 있는 대상은 `lib/generate/pipeline.ts`(생성 파이프라인), `lib/deck-schema.ts`(스키마 검증), 렌더 로직이다.
UI 컴포넌트 전체 테스트는 과하다. 순수 함수·스키마 위주로 소수 정예가 가성비가 가장 높다.

### F-2. 두 개의 생성 시스템 병존

상태: 주의 · 우선순위: 중간

카드뉴스 생성 코드가 두 갈래다.

| 갈래 | 위치 | 사용처 |
|---|---|---|
| 라이브 (A) | `lib/workspace/ai.ts` | 실제 앱 API(`app/api/cards/*`, `app/api/strategy`). **MVP 데모가 쓰는 경로.** |
| 벤치/mock (B) | `lib/generate/*` | `app/api/decks/*` + `scripts/gen` 계열. **라이브 화면은 안 씀.** |

의도된 분리지만(팀 검수는 B로), 처음 보는 사람은 "왜 생성 코드가 두 개지?" 하고 혼란스럽다. 둘이 조금씩 달라지면(drift) 유지보수 부담이 된다.

**방향** — 삭제가 아니라 문서화. ADR 한 장으로 "왜 두 시스템인가 / 각각 언제 쓰나 / 경계는 어디인가"를 못 박는다. 이 분리가 프로젝트의 핵심 설계 결정이므로 ADR 1순위 후보다.

### F-6. 초대형 컴포넌트 — `app/app/create/[id]/page.tsx` (1,135줄)

상태: 주의 · 우선순위: 중간

한 파일이 1,100줄을 넘는다. 카드 생성·편집 화면 전체가 한 컴포넌트에 몰려 있어 읽기·수정·리뷰가 어렵다. 이런 "God component"는 유지보수 난이도의 대표 지표다.

**방향** — 논리 단위(상태 훅 / 하위 UI 섹션)로 분해하되 **동작을 바꾸지 않는 순수 구조 분리만** 한다. F 항목 중 가장 조심해야 할 리팩토링이므로, 테스트(F-1) 안전망을 먼저 깐 뒤 손대는 것을 권장한다.

### F-3. scripts/ 실험 스크립트 정리

상태: 주의 · 우선순위: 중간

`scripts/`에 16개 파일이 있고, 그중 `verify-live-*`(5개)·`measure-card-lifecycle`·`render-only`·`review-card-test` 등은 일회성 실험용이며 package.json에 연결도 안 돼 있다. 어떤 게 "정식 명령"이고 어떤 게 "실험 흔적"인지 구분이 안 돼 첫인상이 어수선하다.

**방향** — 항목마다 셋 중 하나를 적용한다.

- 아직 쓰는 것 → package.json에 명령으로 등록 + 상단 주석으로 용도 명시
- 기록 가치 있는 실험 → `scripts/experiments/`로 이동 + README 한 줄
- 끝난 실험 → 삭제 (git 히스토리에 남으니 안전)

### F-4. lint 경고 8건

상태: 경미 · 우선순위: 낮음

안 쓰는 변수 5건(`liveReady`, `total`, `niche`, `CardNews`, `_h`/`_s`)과 Next 폰트 경고 3건. 기능엔 무해하지만 "다 통과여야 정상"이라는 위생 기준을 흐린다.

**방향** — unused 변수는 제거(리스크 0). 폰트 경고는 Next 권장 방식 이동 여부를 판단(선택).

### F-7. TODO 마커 9개

상태: 경미 · 우선순위: 낮음

대부분 `TODO(Phase5)`·`TODO(인증)`·`TODO(데이터 연결)`처럼 유효한 미래 작업 표시다.

**방향** — 각 TODO를 GitHub Issue로 승격할지 판단한다. 코드 주석 TODO보다 이슈 트래커가 현업 표준이다. 유효한 건 이슈화, 이미 끝난 건 제거.

### F-5. `next lint` deprecated

상태: 경미 · 우선순위: 낮음

Next 16에서 `next lint`가 제거될 예정이라 경고가 출력된다.

**방향** — ESLint CLI로 마이그레이션(공식 codemod 제공). 당장은 아니지만 기록해 둔다.

## 권장 진행 순서

리스크 낮은 것 → 높은 것 순. 각 단계는 별도 PR.

| 순서 | 단계 | 왜 이 자리 |
|---|---|---|
| 1 | F-4 lint 경고 청소 | 리스크 0. 정리 리듬을 익히는 워밍업 |
| 2 | F-3 scripts 정리 | 리스크 낮음. 레포 첫인상 개선 |
| 3 | F-1 테스트 안전망 | Vitest 세팅 + 핵심 로직 소수 테스트. 이후 작업 전체를 지켜줌 |
| 4 | F-2 ADR 문서화 | 포트폴리오 가치 높음. 두 시스템 등 결정 기록 |
| 5 | F-6 초대형 컴포넌트 분해 | 가장 조심. 테스트 깔린 뒤 동작 불변 구조 분리 |
| 6 | F-7 / F-5 마무리 | TODO 이슈화, lint 마이그레이션 |

각 단계 진입 전 함께 확인하고, 끝날 때마다 typecheck·lint·build 통과를 검증한다.
자잘한 QA 수정은 3단계(테스트) 이후에 시작하면 테스트가 그 수정도 지켜준다.
