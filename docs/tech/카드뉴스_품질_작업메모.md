> 마지막 작업일 기준 상태와 다음 할 일을 적어둔다. (다른 노트북에서 이어서 작업할 때 이 문서부터 읽기)

## 배경 — 왜 이 작업을 하나
- 실서비스 카드뉴스가 "깔끔한데 내용이 아쉽다"(뻔하거나 부정확). 예: "비 오는 날 노래 추천" → "조용한 노래 추천" 같은 뻔한 소리, 또는 곡·가수를 **자신 있게 틀리게** 매칭(환각).
- 원인: LLM 단독 생성은 실제 특정 정보(실제 곡·책·맛집) 앞에서 **지어내거나(환각) / 뭉갠다(뻔함)**. 프롬프트만으론 못 벗어남.
- 해법 = 콘텐츠를 3유형으로 나눠 다르게 처리하는 **라우터**:
  - 유형① 공개검증 가능(노래·책·사실) → **그라운딩**(웹서치로 실제 항목 찾아 근거 기반 생성)
  - 유형② 개인·검증 불가(내 맛집·내 사진) → **스캐폴딩**(가짜 대신 "틀+예시" 주고 사용자가 채움)
  - 유형③ 노하우·상식(라떼 만드는 법) → 그냥 생성(모델 지식으로 충분)

## 지금까지 한 것 (실험 하버스 — 라이브 무영향)
전부 스파이크(검증용). 라이브 생성 경로([lib/workspace/ai.ts](../../lib/workspace/ai.ts))는 **아직 안 건드림**. `cardSystemPrompt`에 `export`만 추가함.

- `scripts/compare-card-prompt.ts` (`npm run compare [cafe|finance|reels] [반복수]`)
  - 현재 A 프롬프트 vs B(시스템B 품질규칙 이식) 프롬프트 대조. 결론: 구체적 주제에선 차이 작음.
- `scripts/grounded-card-prototype.ts` (`npm run ground [songs|books]`)
  - 그라운딩 없음 vs 웹서치 그라운딩 대조. **핵심 증명**: 그라운딩 없으면 가수 오귀속(환각), 그라운딩하면 실제 곡+출처.
- `scripts/render-card-comparison.ts` (`npm run render-compare`)
  - "비 오는 날 노래"를 4방식으로 실제 생성 → `out/card-comparison.html`(브라우저로 봄).

## 핵심 결론 (검증됨)
1. **그라운딩이 작동한다.** 웹서치로 실제 곡을 찾아 정확한 카드 생성 확인.
2. **비용 (게시물 1건 = 6슬라이드 통째 기준, 슬라이드당 아님):**
   - 그라운딩 없음: ~45원 (부정확 위험)
   - 그라운딩 Opus: ~261원 (정확)
   - **그라운딩 Sonnet(claude-sonnet-5): ~134원 (정확, 추천)** ← 비용 절반
   - 스캐폴딩: ~41원 (정직, 사용자 수고)
   - 그라운딩은 유형①만 → 맛집·개인경험 게시물은 추가비용 0에 가까움.
3. 웹서치 비용 = $10/1000검색 + 검색결과가 입력토큰으로 주입(지배적 비용). 절감 레버: Sonnet 등 저렴모델 / 동적필터링(`web_search_20260209`, SDK 업글 필요) / 유형 게이팅.

## 이어서 하기 전 준비 (다른 노트북)
1. `git fetch && git checkout fix/cardnews-qa && git pull`
2. `npm install` (새 의존성 있음: @anthropic-ai/sdk 등)
3. `.env.local` **다시 만들기** (gitignore라 전송 안 됨) — 최소 `ANTHROPIC_API_KEY=sk-ant-...` 한 줄. (`.env.example` 참고)
