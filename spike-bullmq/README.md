# BullMQ 내구성 스파이크

> **이 스파이크가 증명하려는 것**
> "코드가 되냐"가 아니라 —
> **배포 환경에서 워커(일꾼 프로세스)가 죽었다 살아나도, 예약된 발행 잡이 사라지지 않고 실행되는가?**
>
> 근거: [Kup_SPEC.md](../docs/product/Kup_SPEC.md) 2.3 발행 메커니즘 / 3.5 예약 발행 —
> *"네이티브 예약이 없으므로 자체 스케줄러가 예약 시각에 워크플로를 트리거"*. 그 "자체 스케줄러"의 내구성 검증.

---

## 용어 (비개발자용)

- **Redis** — 예약("N초 뒤 발행해")을 저장해두는 초고속 창고.
- **BullMQ** — Redis를 이용해 예약 잡을 줄 세우고 시간 되면 꺼내 실행해주는 도구.
- **워커(Worker)** — 줄에서 잡을 꺼내 실제로 실행하는 일꾼 프로세스. ← 죽었다 살리는 대상.
- **핵심 원리** — 예약은 *Redis*에 저장된다. 워커는 그걸 꺼내 쓰는 프로그램일 뿐. 그래서 **워커가 죽어도 Redis만 살아있으면 예약은 안 사라진다.**

## 구조

```
spike-bullmq/
  docker-compose.yml   Redis 한 줄 실행 (AOF 켜서 Redis 재시작도 견딤)
  src/
    connection.js      Redis 연결 (REDIS_URL 환경변수 하나로 로컬↔배포 전환)
    queue.js           예약이 줄 서는 큐
    schedule.js        예약 1건 넣기 (= "N초 뒤 발행", delayed 잡)
    cron.js            반복 스케줄 등록 (= "매일 04:00 인사이트 수집", repeat 잡)
    worker.js          잡을 꺼내 실행 (발행/인사이트 stub, job.name으로 구분)
    status.js          큐 현황 + cron 스케줄러 보기 (예약이 살아있나 눈으로 확인)
```

- **delayed 잡** (schedule.js): "한 번, N초 뒤" — 예약 발행용.
- **cron(repeat) 잡** (cron.js): "패턴마다 자동 반복" — 인사이트 04:00 수집·토큰 갱신용. BullMQ Job Scheduler 사용.

## 실행법

```bash
# 0) 의존성 설치 (최초 1회) — Docker Desktop이 켜져 있어야 함
npm install

# 1) Redis 띄우기
npm run redis:up

# 2) 워커 켜기 (이 창은 켜둔 채로)
npm run worker

# 3) 다른 창에서 예약(delayed) 넣기
npm run schedule -- 10 "테스트 글"   # 10초 뒤 발행 예약

# 3') cron(반복) 등록 — 인사이트 수집형
npm run cron -- "*/10 * * * * *"     # 10초마다(테스트). 워커 창에 반복 실행이 찍힘
npm run cron -- "0 0 4 * * *"        # 운영: 매일 04:00 (Asia/Seoul)
npm run cron -- stop                 # cron 제거

# 큐 현황 + cron 스케줄러 보기 (언제든)
npm run status

# 정리
npm run redis:down     # Redis 끄기 (데이터 보존)
npm run redis:reset    # 데이터까지 완전 초기화
```

## 검증 결과 (2026-06-26, 로컬 Docker)

| 단계 | 내용 | 결과 |
|------|------|------|
| 1 | Docker로 Redis 기동 (`PONG`) | ✅ |
| 2 | 예약 잡이 정해진 시각에 실행 | ✅ 오차 **0.1초** |
| 3 ★ | **워커 강제 종료(crash) → 워커 부재 28초 → 재시작** | ✅ 잡이 Redis에 살아남아 **재시작 즉시 실행** |
| 4 | **cron(repeat) 잡** — `*/5초` 패턴 반복 실행 (2026-06-29) | ✅ :22→:25→:30→:35 자동 반복, delayed와 동시 처리 |

**3단계 상세 (핵심 증명):**
1. 워커를 `taskkill /F`로 강제 종료 (정상 종료 아닌 crash 시뮬레이션)
2. 워커가 **0개인 상태**에서 8초 예약 등록 → `status`: `delayed: 1`
3. 예정 시각이 지나도 (실행할 워커 없음) 잡은 Redis에 **그대로 잔존**
4. 워커 재시작 → **즉시 잡 실행** (예정 1:47:59 / 실제 1:48:28, 죽어있던 28.6초만큼 지연 후 실행)

→ 워커의 생사와 무관하게 예약이 보존됨을 확인. **목표 달성.**

## 실서비스로 갈 때 바뀌는 것

- `worker.js`의 `[발행 STUB 실행]` 자리 → **Meta Graph API 2단계 발행**(컨테이너 생성 → FINISHED 확인 → 게시)으로 교체.
- 이번 스파이크는 그 발행 로직을 **언제·확실히 트리거하느냐(스케줄링 내구성)**만 검증한 것.
